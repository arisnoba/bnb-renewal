# 보안 운영 인수인계

> 목적: 배포 후 운영자가 보안 설정, 비공개 문의 첨부, 의존성 점검을 같은 기준으로 확인할 수 있게 한다.
> 마지막 갱신: 2026-07-14

## 범위와 보류 사항

이 문서는 현재 코드에 구현된 보안 통제와 운영 점검 절차만 다룬다.

문의와 첨부파일의 보유기간 및 정기 삭제 주기는 아직 확정하지 않았다. 개인정보처리방침이 확정되기 전에는 이 문서를 근거로 자동 삭제 작업을 만들거나 기존 문의·첨부를 삭제하지 않는다. 정책 확정 후 처리방침 문구, Payload 데이터, 비공개 R2 객체의 삭제 시점을 같은 작업에서 맞춘다.

## 1. 환경변수와 소유권

실제 값은 저장소나 문서에 기록하지 않는다. Vercel Production과 Preview에 각각 설정하고, 클라이언트 소유 계정에서 관리한다.

| 구분 | 환경변수 | 운영 기준 |
| --- | --- | --- |
| Payload 인증 | `PAYLOAD_SECRET` | Production과 Preview에 서로 다른 충분히 긴 랜덤 값을 설정한다. 누락 시 운영·미리보기 서버가 시작되지 않는 것이 정상이다. |
| DB | `DATABASE_URL`, `DATABASE_URL_UNPOOLED` | Vercel 런타임은 pooled URL을 우선하고, migration·백업은 direct URL을 확인한다. |
| DB 연결 수 | `PAYLOAD_DATABASE_POOL_MAX` | 선택값이다. Vercel 기본값은 3이며 장애 근거 없이 늘리지 않는다. |
| Turnstile | `NEXT_PUBLIC_TURNSTILE_SITE_KEY`, `TURNSTILE_SECRET_KEY` | 공개 키와 비밀 키를 혼동하지 않는다. 비밀 키는 서버 환경에만 둔다. |
| 공개 R2 | `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET`, `R2_ENDPOINT`, `R2_PUBLIC_BASE_URL` | 공개 media 전용 bucket과 token을 사용한다. |
| 비공개 R2 | `R2_PRIVATE_ACCESS_KEY_ID`, `R2_PRIVATE_SECRET_ACCESS_KEY`, `R2_PRIVATE_BUCKET` | 문의 첨부 전용 비공개 bucket과 최소 권한 token을 사용한다. 현재 S3 endpoint는 `R2_ENDPOINT`를 공유한다. |

## 2. HTTP 보안 설정

보안 헤더의 코드 정본은 [`config/security-headers.mjs`](../config/security-headers.mjs)다. [`next.config.mjs`](../next.config.mjs)가 모든 경로에 적용한다.

현재 애플리케이션 헤더:

- `Content-Security-Policy`
- `Permissions-Policy: camera=(), geolocation=(), microphone=()`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`

HSTS는 Vercel 플랫폼 설정을 유지한다. CSP를 바꿀 때는 Cloudflare Turnstile, 네이버 지도, YouTube, Vercel Insights, Payload 관리자 화면을 함께 확인한다.

배포 후 확인:

```bash
curl -I https://<운영-도메인>/
curl -I https://<운영-도메인>/admin
```

응답 헤더를 확인한 뒤 브라우저 콘솔에서 CSP 차단이 없는지 확인한다. 허용 origin을 넓힐 때는 해당 서비스의 정확한 origin만 추가하고 `*`를 사용하지 않는다.

## 3. 공개 API와 Payload 표면

- Payload REST 조회 최대 depth는 [`payload.config.ts`](../payload.config.ts)의 `maxDepth: 3`으로 제한한다.
- GraphQL 라우트는 등록하지 않는다. `/api/graphql`, `/graphql`이 404인 상태가 현재 기준이다.
- 비로그인 Payload 읽기는 각 컬렉션의 공개 상태 조건을 통과한 문서만 허용한다.
- 상담 API는 IP당 10분 5회, Turnstile API는 IP당 1분 20회, 캐스팅 현황 추가 조회는 IP당 1분 60회로 제한한다.
- rate-limit 카운터는 PostgreSQL `payload_kv`에 저장하며 식별자는 `PAYLOAD_SECRET` 기반 HMAC-SHA-256 값만 저장한다.
- rate-limit 저장소 장애 시 요청을 무제한 허용하지 않고 503을 반환한다.

## 4. 비공개 문의 첨부

### 저장 기준

- 신규 첨부는 공개 media bucket이 아니라 `R2_PRIVATE_BUCKET`에 저장한다.
- DB에는 공개 URL이 아니라 `inquiries/attachments/...` object key와 원본 파일명만 저장한다.
- 한 문의당 첨부 1개, 최대 4MB다.
- 허용 형식은 JPG/JPEG, PNG, PDF다. 확장자, MIME, 실제 파일 시그니처가 모두 일치해야 한다.
- 업로드 후 문의 DB 저장이 실패하면 방금 올린 비공개 객체 삭제를 시도한다.

### 다운로드 기준

관리자 다운로드 경로는 `/api/inquiries/{id}/attachment`다.

- Payload 로그인이 필요하다.
- 현재 사용자에게 `inquiries` 문서 읽기 권한이 있어야 한다.
- `overrideAccess: false`를 유지한다.
- 응답은 `Cache-Control: private, max-age=0, no-store`와 `Content-Disposition: attachment`를 사용한다.
- 비공개 R2 설정이 없으면 503, 객체가 없거나 문서 접근이 불가능하면 404를 반환한다.

확인 코드:

- 업로드: [`src/app/(payload)/api/consult/route.ts`](<../src/app/(payload)/api/consult/route.ts>)
- 다운로드: [`src/app/(payload)/api/inquiries/[id]/attachment/route.ts`](<../src/app/(payload)/api/inquiries/[id]/attachment/route.ts>)
- 파일 정책: [`src/lib/inquiryAttachment.ts`](../src/lib/inquiryAttachment.ts)
- 시그니처 검사: [`src/lib/uploadFileValidation.ts`](../src/lib/uploadFileValidation.ts)

### 기존 공개 첨부 이전

먼저 읽기 전용 계획을 확인한다.

```bash
npm run payload:migrate-inquiry-attachments:plan
```

비공개 bucket 설정과 대상을 확인한 뒤 복사한다.

```bash
npm run payload:migrate-inquiry-attachments:write
```

공개 원본 삭제는 비공개 복사, 파일 크기 일치, 관리자 다운로드 확인을 끝낸 뒤 별도 승인을 받아 실행한다.

```bash
ALLOW_DESTRUCTIVE_C0=1 npm run payload:migrate-inquiry-attachments:write -- --delete-public
```

삭제 후 확인:

1. 관리자 인증 다운로드가 성공한다.
2. 비로그인 요청은 401이다.
3. DB의 `attachment_url`은 비어 있고 `attachment_object_key`만 남는다.
4. 비공개 object는 읽을 수 있고 공개 bucket의 같은 key는 존재하지 않는다.

## 5. 의존성과 품질 점검

lockfile 기준 설치부터 확인한다.

```bash
npm ci
npm ls
```

운영 의존성 취약점은 아래 명령을 배포 전과 직접 의존성 변경 시 실행한다.

```bash
npm audit --omit=dev
```

전체 audit의 개발 도구 취약점은 운영 결과와 분리해 기록한다. `npm audit fix --force`는 Payload·Next.js·Vercel CLI의 호환성을 깨뜨릴 수 있으므로 자동 실행하지 않는다. Payload 패키지군은 같은 버전으로 맞추고, 변경 후 lockfile과 peer dependency 상태를 확인한다.

기본 품질 검증:

```bash
npm test
npm run typecheck
npm run lint
npm run build
```

GitHub Actions의 [Code quality](../.github/workflows/quality.yml)는 Node.js 22에서 `npm ci`, 테스트, 타입체크, 린트를 실행한다. 빌드는 로컬 또는 배포 단계에서 별도로 확인한다.

## 6. 운영 점검 주기

### 배포 또는 환경변수 변경 시

- Production과 Preview의 `PAYLOAD_SECRET` 존재 여부를 확인한다.
- 공개·비공개 R2 token이 서로 다른 bucket 범위인지 확인한다.
- 상담 제출, Turnstile, 관리자 첨부 다운로드를 실제로 확인한다.
- 운영 도메인의 보안 헤더와 브라우저 CSP 오류를 확인한다.
- `npm audit --omit=dev`와 기본 품질 검증 결과를 남긴다.

### 계정 또는 token 교체 시

- 새 token을 먼저 등록하고 읽기·쓰기 스모크 테스트를 통과시킨다.
- Vercel Production과 Preview를 각각 재배포한다.
- 기존 token은 새 배포 검증이 끝난 뒤 폐기한다.
- DB URL이나 R2 endpoint를 문서·이슈에 원문으로 남기지 않는다.

## 7. 인수인계 체크리스트

- [ ] Vercel, Neon, Cloudflare 계정과 결제 주체가 클라이언트 소유인지 확인했다.
- [ ] Production과 Preview 환경변수 담당자와 갱신 절차를 전달했다.
- [ ] Payload 관리자 역할별 권한과 문의 첨부 다운로드를 확인했다.
- [ ] 공개 R2와 비공개 문의 첨부 bucket의 용도를 구분해 전달했다.
- [ ] 보안 헤더와 rate-limit의 코드 정본 위치를 전달했다.
