# 레거시 마이그레이션 정책

> 목적: 레거시 DB, FTP 이미지, Payload/Postgres/R2 이관 기준을 한 문서로 관리한다.
> 마지막 갱신: 2026-05-20

## 기본 방향

현재 이관 기준은 과거의 P0/P1/P2 또는 C0 단계가 아니다. 해당 기준은 폐기된 계획으로 보고 현재 작업 판단에 사용하지 않는다.

현재 기준 파이프라인은 아래 하나다.

```text
data/legacy_dumps
  -> 로컬 MariaDB 복원
  -> 로컬 MariaDB work table 구성
  -> 로컬 Postgres/Payload DB에 정제 적재
  -> 로컬 검증
  -> Vercel Neon으로 migration/데이터 이전
```

대량 write는 dry-run과 샘플 검증 후 명시적으로만 진행한다.

## 데이터 흐름

```text
data/legacy_dumps의 센터별 MariaDB dump
  -> 로컬 MariaDB 복원
  -> bnb_legacy_work.* work table
  -> FTP/legacy 이미지 dry-run
  -> public/legacy/* staging/cache
  -> 로컬 Postgres/Payload DB 정제 적재
  -> Payload media relation 또는 R2 objectKey 연결
  -> 로컬 검증
  -> Vercel Neon으로 migration/데이터 이전
  -> 개발자 계정 R2 업로드/검증
  -> 클라이언트 계정 R2로 동일 objectKey 복사
```

## 원본 보존 원칙

MariaDB work table과 `public/legacy`는 최종 서비스 경로가 아니라 원본 추적용 staging/cache다.

로컬 Postgres/Payload DB에는 레거시 테이블 컬럼을 그대로 복제하지 않는다. 최종 모델은 관리자와 공개 사이트에서 쓰는 의미 있는 운영 필드로 정제하고, 원본 재추적에 필요한 최소 메타데이터만 숨김 또는 레거시/원본 영역에 둔다.

Payload에 최소로 보존할 값:

- `sourceDb`
- `sourceTable`
- `sourceId`
- 필요한 경우 `legacyUrl`
- 필요한 경우 `legacyPath`
- 필요한 경우 `legacyMeta`

MariaDB work table과 asset staging에서 보존해야 하는 값:

- `source_db`
- `source_table`
- `source_id`
- `bo_table`
- 원본 파일명
- 원본 FTP/legacy path
- 다운로드된 `public/legacy` path
- `legacy_meta`

원본 DB, work table, legacy path는 임의로 덮어쓰지 않는다.

## 날짜 이관 원칙

레거시 게시판 계열 데이터에서 `published_at`은 콘텐츠가 공개/게시된 날짜이고, Payload 시스템 필드 `createdAt`은 관리자 목록의 `등록일` 정렬 기준으로도 쓰인다.

레거시 work table에 `published_at`과 `created_at`이 있는 컬렉션은 Payload 적재 시 두 값을 모두 전달한다. 원본에서 별도 생성일을 신뢰할 수 없거나 기존 게시판 `wr_datetime`이 게시/생성 기준이면 `created_at`은 `published_at`과 같은 값으로 둔다.

이미 로컬 Postgres에 적재된 레거시 문서에서 `created_at`이 적재 시각으로 들어간 경우, 레거시 추적값이 있는 문서만 대상으로 `created_at = published_at` 보정 migration을 작성한다.

기준:

- 대상은 `source_id IS NOT NULL` 같은 레거시 추적값이 있는 문서로 제한한다.
- `published_at IS NOT NULL`인 문서만 보정한다.
- 신규 관리자 작성 문서나 일반 Payload 시스템 문서까지 일괄 수정하지 않는다.
- 관리자 날짜 표시는 `발행일`과 `등록일`은 날짜만, `수정일`은 시간까지 노출한다.

## 이미지 이관 원칙

- `public/legacy`는 R2 업로드 전 로컬 cache로만 본다.
- 서비스용 이미지는 Payload `media` relation 또는 R2 `objectKey`로 연결한다.
- DB/Payload에는 개발자 R2 완성 URL을 저장하지 않는다.
- 대표 이미지, 본문 이미지, 로고, 학생 이미지처럼 역할을 분리한다.
- 대량 write와 대량 media 생성은 dry-run 결과 확인 후 명시적으로만 수행한다.
- R2는 실제 디렉터리가 아니라 object key prefix를 디렉터리처럼 보여준다. 업로드 key는 `media/{컬렉션}/{역할}/파일명`처럼 나눠서 저장한다.
- media 생성 스크립트에서 `filename`을 `direct-castings-body-11-874bfd79b6-...jpg`처럼 평탄하게 만드는 방식은 금지한다. 최종 R2 object key는 `media/{컬렉션}/{역할}/{source_id}/{원본파일명}`처럼 `/`로 구분된 계층을 가져야 한다.
- 정제 후 업로드에서는 원본 테이블명 세그먼트를 넣지 않는다. 컬렉션/역할 prefix와 정제된 row id만으로 구분한다.
- 정제 후 R2 파일명은 원본 긴 파일명을 그대로 쓰지 않는다. `{role-base}-{id}.{ext}`, `{role-base}-{id}-{size}.{ext}`처럼 짧은 이름을 사용하고, 원본 추적은 마이그레이션 리포트에 남긴다.
- `prefix`와 `filename`을 합쳐 최종 key를 만드는 구조라면, `prefix`에는 컬렉션/역할까지만 넣고 이후 id/파일명은 공통 objectKey 생성 helper가 붙인다.
- 대표 이미지와 썸네일도 기본값은 원본 1개 업로드다. Payload `thumbnail`, `square`, `small`, `medium`, `large`, `xlarge`, `og` 파생본은 명시적으로 필요한 경우에만 만든다.
- R2 업로드 스크립트에서 원본이 2MB를 넘으면 우선 원본을 재인코딩해 줄이고, 변형 파일 생성으로 스토리지를 늘리는 방식은 기본값으로 쓰지 않는다.
- 본문 이미지(`*.body-image`, `media/{collection}/body-images`)는 본문 렌더링에 필요한 원본만 R2에 업로드한다. Payload size 파생본(`thumbnail`, `square`, `og` 등)은 업로드하지 않고 `sizes_*_filename`을 `NULL`로 비워, API가 존재하지 않는 size URL을 내려주지 않게 한다.
- 본문 이미지 파일명은 한 문서 안의 순서를 포함한다. 예: `media/direct-castings/body-images/3/direct-casting-body-image-3-1.png`, `media/direct-castings/body-images/3/direct-casting-body-image-3-2.png`.
- 본문 이미지 R2 업로드 대상은 현재 Lexical `body`의 upload node가 실제 참조하는 `media.id`만으로 제한한다. 같은 prefix 아래에 남아 있는 미참조 media row는 별도 cleanup 대상이며, 본문 업로드 배치에 섞지 않는다.
- media relation 백필 전후 리포트에는 최소 3개 이상의 최종 object key 샘플과 `prefix`, `filename` 분포를 남겨, R2 콘솔에서 폴더별 구분이 가능한지 검증한다.
- 로컬 개발/마이그레이션 검수 중에는 `public/media`에 파일이 있으면 `/api/media/file/...`에서 로컬 파일을 먼저 서빙할 수 있다. 프로덕션 기준에서는 이 fallback에 의존하지 않고 R2 object가 존재해야 한다.
- 프로덕션 전환 전에는 연결된 Payload `media.prefix + filename`이 R2에 실제 업로드되어 있고 공개 URL이 200으로 응답하는지 확인한다.
- 새 R2 업로드 스크립트는 `--media-role <collection.asset-role>`을 우선 사용한다. 허용 role과 prefix는 `src/lib/r2ObjectKeys.ts`의 `R2_MEDIA_PREFIX_BY_ROLE`에서 관리한다.
- `scripts/payload-migration/upload-images.ts`는 dry-run/write 리포트에 `prefix`, `mediaRole`, `objectKeySamples`를 남긴다. `--list-media-roles`로 현재 허용 목록을 확인한다.

본문 이미지 R2 업로드 검증 기준:

- 업로드 전: R2 대상 prefix의 기존 partial/stale object 수를 확인한다.
- 기존 partial 업로드가 있으면 대상 prefix만 삭제하고 다시 시작한다.
- 업로드 후: R2 object count와 현재 body upload ref 수가 일치해야 한다.
- 로컬 Postgres/Payload DB에서 대상 media row의 `prefix`는 `media/{collection}/body-images/{source_id}`, `filename`은 짧은 원본 파일명이어야 한다.
- 대상 media row의 `sizes_thumbnail_filename`, `sizes_square_filename`, `sizes_small_filename`, `sizes_medium_filename`, `sizes_large_filename`, `sizes_xlarge_filename`, `sizes_og_filename`은 모두 `NULL`이어야 한다.
- 공개 URL 샘플은 `HEAD` 또는 `GET`으로 200 응답과 올바른 content-type을 확인한다.

현재 적용된 media 연결:

- `news.thumbnailPath` -> `thumbnailMedia`, `meta.image` dry-run 스크립트
- `artist-press.thumbnailPath` -> `thumbnailMedia`, `meta.image` dry-run 스크립트
- `artist-press.agencyLogoPath` -> `agencyLogoMedia` dry-run 스크립트
- `exam-school-logos` 다운로드 리포트 -> `logoMedia` 실제 연결 완료
- media 생성 스크립트의 신규 R2 prefix 기준: `media/{collection}/{asset-role}`. 예: `media/news/thumbnails`, `media/news/body-images`, `media/artist-press/thumbnails`, `media/artist-press/agency-logos`, `media/artist-press/body-images`, `media/agencies/logos`, `media/casting-directors/profile-images`, `media/exam-school-logos/logos`, `media/direct-castings/thumbnails`, `media/direct-castings/body-images`

## 주요 명령

```bash
npm run legacy:db:import
npm run legacy:db:verify
npm run legacy:work:news
npm run legacy:work:artist-press
npm run payload:legacy:link-news-thumbnails
npm run payload:legacy:link-artist-press-media
npm run payload:legacy:link-exam-school-logo-media
npm run payload:legacy:upload-images:r2 -- --list-media-roles
npm run payload:legacy:upload-images:r2 -- --dry-run --media-role news.thumbnail --input path/to/scan.json --output docs/reports/r2-news-thumbnail-dry-run.json
```

실제 DB 쓰기는 각 스크립트의 `--write` 옵션이 있는 경우에만 수행한다.

## 로컬 Postgres에서 Neon으로 이전

정제와 검증은 로컬 Postgres/Payload DB에서 먼저 끝낸다. Vercel Neon은 로컬에서 검증된 schema와 데이터를 옮기는 대상으로 본다.

기준:

- 로컬 MariaDB는 원본 복원과 work table 생성용이다.
- 로컬 Postgres는 Payload 컬렉션 기준으로 정제된 데이터의 1차 정본이다.
- Neon 이전 전에는 schema diff, 샘플 레코드, media relation, legacy fallback을 확인한다.
- Neon에는 검증된 migration과 데이터만 반영한다.
- 운영 R2 이전과 Neon 이전은 분리해서 검증한다.

## Migration 주의

Payload migration 생성 전에는 현재 DB schema diff를 먼저 확인한다. Payload가 rename/create를 물을 때 자동 선택하면 기존 컬럼을 잘못 rename할 수 있다.

특히 레거시 필드는 삭제하지 않는다. 신규 UX 필드는 optional로 추가하고, 프론트는 신규 필드 우선 fallback으로 렌더링한다.
