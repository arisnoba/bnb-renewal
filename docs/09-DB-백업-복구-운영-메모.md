# DB 백업 복구 운영 메모

작성일: 2026-07-01
마지막 갱신: 2026-07-20

## 목적

이 문서는 납품 후 운영 중 DB 사고가 발생했을 때 최소 복구 기준을 정리한다.

현재 기본 목표는 Neon 유료 플랜 없이 `원격 Vercel Neon DB`의 일일 백업을 외부 저장소에 15일간 유지하는 것이다. 일일 백업의 복구 시점 목표(RPO)는 최대 24시간이며, Neon Free가 제공하는 짧은 restore window는 보조 수단으로 사용한다.

## 전제

- 대상 DB는 `원격 Vercel Neon DB`다.
- Vercel은 앱 호스팅과 Marketplace 연동을 담당한다.
- PostgreSQL 백업/복구의 실제 제공자는 Neon이다.
- 외부 백업 저장소는 Cloudflare R2 비공개 bucket `baewoo-private`다.
- 문의 첨부와 DB 백업이 같은 bucket을 쓰므로 object key prefix와 Lifecycle 적용 범위를 반드시 분리한다.
- 납품 프로젝트에서는 개인 NAS, 개인 로컬 디스크, 개발자 개인 계정의 백업을 운영 백업 기준으로 삼지 않는다.
- R2 media object는 DB 백업만으로 복구되지 않는다. R2 파일 삭제나 손상까지 사고 범위에 포함하면 별도 R2 보존 정책이 필요하다.

## 권장 운영 기준

### 1. GitHub Actions 일일 pg_dump

현재 상시 운영 백업은 [database-backup.yml](../.github/workflows/database-backup.yml)이 담당한다.

운영 기준:

- 매일 02:17 KST에 실행하고 필요할 때 `workflow_dispatch`로 수동 실행한다.
- pooled URL이 아니라 GitHub Environment secret `DATABASE_URL_UNPOOLED`을 사용한다.
- PostgreSQL 17 client로 전체 DB를 custom format(`pg_dump -Fc`)으로 압축한다.
- `pg_restore --list` 결과에 복원 항목이 있는지 확인하고 SHA-256 체크섬을 생성한다.
- dump와 checksum만 비공개 R2에 업로드하고 GitHub artifact에는 DB 파일을 남기지 않는다.
- 업로드 후 R2 object의 크기와 checksum object 존재 여부를 확인한다.

저장 경로:

```text
baewoo-private/
  database-backups/
    production/
      YYYY/MM/DD/
        bnb-renewal-YYYYMMDDTHHMMSSZ.dump
        bnb-renewal-YYYYMMDDTHHMMSSZ.dump.sha256
```

필요한 GitHub Environment secrets:

- `DATABASE_URL_UNPOOLED`
- `R2_ENDPOINT`
- `R2_BACKUP_ACCESS_KEY_ID`
- `R2_BACKUP_SECRET_ACCESS_KEY`

R2 backup token은 가능하면 `baewoo-private`에만 접근하는 전용 token으로 발급한다. R2 token은 prefix 단위로 제한되지 않으므로 workflow에서 `database-backups/production/` 외의 key를 사용하지 않도록 고정 검사를 유지한다.

### 2. R2 Lifecycle 15일

백업 정리는 GitHub Actions의 삭제 명령이 아니라 R2 Lifecycle에 맡긴다.

운영 기준:

- bucket: `baewoo-private`
- rule name: `Expire production database backups after 15 days`
- prefix: `database-backups/production/`
- expiration: 15일
- 문의 첨부가 저장되는 `inquiries/attachments/`에는 이 규칙을 적용하지 않는다.
- 규칙 변경 전후에 Cloudflare Dashboard 또는 Wrangler로 prefix와 만료 일수를 다시 확인한다.

2026-07-20 기준 위 규칙을 실제 적용하고, 기존 `Default Multipart Abort Rule`과 함께 활성화된 상태를 확인했다. 기존 규칙과 문의 첨부 prefix는 변경하지 않았다.

확인 명령:

```bash
npx wrangler r2 bucket lifecycle list baewoo-private
```

### 3. Neon Free restore window

Neon Free의 restore window는 짧으므로 일일 외부 백업을 대체하지 않는다. 사고 시점이 Neon Free의 복원 범위 안에 있으면 해당 기능을 먼저 검토하고, 범위를 벗어나면 R2의 dump를 사용한다.

유료 전환 후에는 다음 구성을 추가할 수 있다.

- Neon Launch restore window 7일
- 일일 Neon snapshot 7~14일
- 복원 시 운영 DB를 바로 덮어쓰기보다 새 branch에서 먼저 확인

참고 기준:

- Neon Launch 플랜은 최대 7일, Scale 플랜은 최대 30일까지 restore window를 둘 수 있다.

### 4. 위험 작업 직전 수동 백업

수동 `pg_dump`는 상시 운영 백업이 아니라 위험 작업 직전의 이벤트성 백업으로 사용한다.

사용 시점:

- 원격 migration 전
- 대량 데이터 수정 전
- R2 path, media relation 등 되돌리기 어려운 데이터 보정 전
- 납품 전 최종 상태 보관이 필요한 경우

권장 형식:

```bash
pg_dump "$DATABASE_URL_UNPOOLED" -Fc -Z 9 > neon-neondb-YYYYMMDDTHHMMSSZ-pre-change.dump
pg_restore -l < neon-neondb-YYYYMMDDTHHMMSSZ-pre-change.dump > neon-neondb-YYYYMMDDTHHMMSSZ-pre-change.dump.list
shasum -a 256 neon-neondb-YYYYMMDDTHHMMSSZ-pre-change.dump > neon-neondb-YYYYMMDDTHHMMSSZ-pre-change.dump.sha256
```

주의:

- 운영 DB와 같은 PostgreSQL 17 `pg_dump`/`pg_restore` client를 사용한다.
- 로컬 Homebrew 클라이언트가 오래된 경우 `postgres:17-alpine` 같은 컨테이너 클라이언트를 사용한다.
- 운영 기준 백업은 클라이언트 소유 계정의 Neon 또는 클라이언트 소유 cloud storage에 있어야 한다.

## 납품 시 확인 항목

- [ ] Vercel 프로젝트가 클라이언트 소유 팀 또는 계정으로 이전됐다.
- [ ] Neon 프로젝트가 클라이언트 소유 조직 또는 계정에 있다.
- [ ] GitHub `Production – bnb-renewal` Environment에 백업용 secrets가 등록됐다.
- [ ] `Daily Neon database backup` workflow를 수동 실행해 성공했다.
- [ ] R2에 dump와 checksum object가 생성되고 크기가 0보다 크다.
- [ ] `database-backups/production/` prefix에만 15일 Lifecycle이 적용됐다.
- [ ] 월 1회 최신 dump를 임시 로컬 Postgres에 실제 복원한다.
- [ ] 유료 전환 시 Neon restore window와 snapshot 정책을 다시 결정한다.
- [ ] DB 백업 기준과 R2 media 보존 기준이 별도 항목임을 인수인계 문서에 명시했다.

## 월간 복원 검증

백업 파일 생성과 `pg_restore --list` 확인만으로 실제 복원이 보장되지는 않는다. 월 1회 최신 dump와 checksum을 Cloudflare Dashboard에서 내려받아 임시 로컬 DB에 복원한다.

```bash
shasum -a 256 -c bnb-renewal-YYYYMMDDTHHMMSSZ.dump.sha256
docker compose exec -T postgres createdb -U postgres bnb_renewal_restore
docker compose exec -T postgres pg_restore \
  -U postgres \
  -d bnb_renewal_restore \
  --no-owner \
  --no-privileges \
  < bnb-renewal-YYYYMMDDTHHMMSSZ.dump
docker compose exec -T postgres psql \
  -U postgres \
  -d bnb_renewal_restore \
  -c 'SELECT COUNT(*) FROM payload_migrations;'
```

이미 `bnb_renewal_restore`가 있으면 새 검증용 DB 이름을 사용한다. 운영 DB URL에는 직접 복원하지 않는다. 검증용 DB 정리는 데이터 확인을 마친 뒤 별도 삭제 대상으로 처리한다.

## 사고 대응 기준

### 데이터 오입력 또는 일부 레코드 삭제

1. 사고 발생 시각과 정상 기준 시각을 기록한다.
2. 사고 시점이 Neon restore window 안이면 해당 시점의 별도 branch 복원을 검토한다.
3. 범위를 벗어나면 사고 전 R2 dump를 임시 로컬 Postgres에 복원한다.
4. 운영 DB와 복원 DB의 차이를 비교한다.
5. 필요한 row만 선별 복구한다.
6. 복구 후 Payload admin과 공개 route에서 실제 데이터를 확인한다.

### migration 또는 대량 보정 실패

1. 변경 직전 수동 backup 또는 snapshot 존재 여부를 확인한다.
2. 가능하면 운영 DB를 직접 되돌리지 말고 복원 branch에서 먼저 검증한다.
3. schema/data가 함께 깨졌으면 `pg_restore` 또는 Neon restore를 검토한다.
4. 복원 후 `payload_migrations`, 주요 collection row count, 공개 페이지 대표 route를 확인한다.

### R2 media 삭제 또는 손상

DB 복구만으로는 R2 object가 되살아나지 않는다.

확인 항목:

- DB의 media row가 존재하는지
- media row의 `objectKey`가 남아 있는지
- `R2_PUBLIC_BASE_URL + objectKey`가 실제로 `200` 응답을 주는지

R2 파일까지 복구해야 하는 운영 요건이 있으면 Cloudflare R2 쪽 보관, 복제, 별도 object backup 정책을 추가로 설계해야 한다.

## 현재 프로젝트 권장안

현재 프로젝트의 최소 납품 운영 기준은 아래 조합으로 둔다.

```text
Neon Free
+ GitHub Actions 일일 pg_dump
+ baewoo-private/database-backups/production/ 업로드
+ R2 Lifecycle 15일
+ 위험 작업 직전 수동 pg_dump
+ 월 1회 임시 로컬 Postgres 복원 테스트
```

유료 전환 시에는 `Neon Launch + Restore Window 7일`을 위 구성에 추가한다.

NAS나 개발자 개인 로컬 백업은 운영 백업 기준에서 제외한다. 개발 중 생성한 로컬 덤프는 작업자 참고용 또는 일회성 안전장치일 뿐, 납품 후 복구 책임을 충족하는 백업으로 보지 않는다.
