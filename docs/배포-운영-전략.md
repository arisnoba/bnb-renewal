# 배우앤배움 통합 사이트 배포/운영 전략

> 작성일: 2026-04-01
> 목적: 의사결정자 검토용 배포안 비교, 비용/운영 리스크 정리, 실시간 콘텐츠 반영 방식 정의
> 연관 문서: `docs/SEO-URL-전략.md`

## 1. 결론 요약

배우앤배움 통합 사이트는 `Vercel 기반 통합 운영안`이 가장 적합하다.

이유는 3가지다.

- SEO 목표가 높다. 센터별 상세 URL, 메타데이터, sitemap, 301 이전까지 안정적으로 운영해야 한다.
- 관리자에서 수정하는 콘텐츠가 많다. 특히 `공지사항`, `캐스팅`, `배우/강사`는 수정 후 빠르게 공개 사이트에 반영되어야 한다.
- 클라이언트가 비용에 민감하다. 이 경우에도 `매 요청 SSR`보다 `ISR + on-demand revalidation` 구조를 쓰면 Vercel 비용을 통제하면서 실시간성에 가깝게 운영할 수 있다.

권장 구조는 아래와 같다.

- 공개 사이트 + 관리자(Payload) + API: `동일한 Next.js 앱`
- 앱 호스팅: `Vercel`
- DB: `Vercel Marketplace Postgres(예: Neon)` 또는 별도 Postgres
- 파일 업로드: `Vercel Blob` 또는 S3 계열
- 콘텐츠 즉시 반영: `Payload 저장 후 revalidatePath()`

즉, `별도 Payload 서버를 하나 더 둬야만 하는 구조`는 아니다.

다만 `DB`와 `파일 스토리지`는 필요하다. 이것은 Payload 때문이 아니라 CMS 자체가 데이터를 저장해야 하기 때문이다.

## 2. 의사결정 포인트

이번 배포안은 아래 질문 4개로 판단하면 된다.

| 질문                                                       |    중요도 | 의미                                        |
| ---------------------------------------------------------- | --------: | ------------------------------------------- |
| 관리자 수정 후 공개 사이트에 바로 반영되어야 하는가        | 매우 높음 | 공지/캐스팅/배우 상세 운영에 직접 영향      |
| SEO용 HTML을 검색엔진이 처음부터 읽을 수 있어야 하는가     | 매우 높음 | 상세 본문, meta, canonical, schema, sitemap |
| 운영 인력이 적고 배포/장애 대응을 단순하게 가져가야 하는가 |      높음 | Vercel 쪽이 유리                            |
| 월 고정비보다 예측 가능한 최소 운영비가 중요한가           |      높음 | SSR 남발은 피하고 ISR 중심으로 설계 필요    |

배우앤배움은 위 4개 모두 `예`에 가깝다.

## 2-1. SSG와 SEO에 대한 오해 정리

### 핵심 정리

`SSG 자체가 SEO에 불리한 것은 아니다.`

SSG도 빌드 시점에 페이지 본문, 제목, 메타데이터, 구조화 데이터가 HTML 안에 완성되어 있으면 SEO에 유리하다.

문제가 되는 경우는 아래와 같다.

- HTML은 껍데기만 정적으로 배포하고
- 실제 공지사항/캐스팅/배우 데이터는 브라우저가 나중에 DB/API에서 읽는 구조

이 경우 검색엔진 봇이 처음 받는 HTML에는 본문이 비어 있거나 약할 수 있다. 이 패턴이 SEO에 불리한 것이다.

즉, 이번 프로젝트의 쟁점은 `SSG냐 아니냐`가 아니라 아래 두 가지다.

- 콘텐츠가 `HTML 생성 시점`에 포함되는가
- 관리자 수정 후 `재빌드 없이` 반영되어야 하는가

### 배우앤배움에 적용하면

- 고정 페이지는 `SSG`로 충분하다
- 공지사항/캐스팅/배우 상세처럼 관리자가 자주 수정하는 페이지는 `SSG만으로는 부족`하다
- 따라서 해당 영역은 `ISR`을 기본으로 하고, 필요한 경우 일부만 `SSR`을 사용한다

### 의사결정자 설명용 문장

`정적 페이지(SSG)는 HTML 안에 콘텐츠가 미리 들어가 있으면 SEO에 문제가 없습니다. 다만 공지사항, 캐스팅, 배우/강사 정보처럼 관리자에서 수시로 수정하는 콘텐츠는 SSG만으로 운영하면 재빌드 전까지 반영되지 않으므로, 해당 페이지는 ISR/SSR 기반으로 운영해야 합니다.`

## 3. 배포안 2가지

### 안 A. Vercel 통합형

구조:

- 하나의 Next.js 앱 안에 공개 사이트와 Payload CMS를 함께 운영
- Vercel에 앱 배포
- DB는 Vercel Marketplace Postgres 또는 외부 Postgres 사용
- 업로드 파일은 Vercel Blob 또는 외부 Object Storage 사용

개념도:

```text
사용자/관리자
    ↓
 Vercel (Next.js 앱)
    ├─ 공개 사이트
    ├─ Payload 관리자
    ├─ REST/GraphQL/Local API
    └─ revalidatePath / ISR
    ↓
 Postgres + Blob Storage
```

핵심 특징:

- 운영 창구를 사실상 Vercel 중심으로 단순화 가능
- SEO 친화적 URL과 서버 렌더링/재검증 구조에 적합
- 관리자 수정 후 필요한 경로만 다시 생성 가능
- 정적 페이지와 동적 페이지를 혼합 운영 가능

### 안 B. 일반 웹호스팅 정적 업로드형 + 별도 CMS

구조:

- 공개 사이트는 `next export` 또는 유사 정적 빌드 후 웹호스팅에 업로드
- Payload CMS는 별도 Node 실행 환경에 배포
- DB와 파일 스토리지는 별도 운영

개념도:

```text
사용자
    ↓
 일반 웹호스팅 (정적 HTML/CSS/JS)

관리자
    ↓
 별도 Payload 서버
    ↓
 Postgres + Blob Storage
```

핵심 특징:

- 공개 사이트 파일 배포만 보면 단순해 보임
- 하지만 CMS, DB, 파일 저장소는 결국 별도 운영이 필요
- 실시간 반영 요구와 SEO를 동시에 만족시키기 어려움
- 장애 지점이 오히려 늘어날 수 있음

## 4. "Vercel 하나만" 가능한가?

### 답변

`운영 관점에서는 거의 가능하다.`

정확히 말하면:

- `Payload 앱 서버를 따로 두지 않아도 된다`
- 공개 사이트와 Payload CMS를 `같은 Next.js 앱`, `같은 Vercel 프로젝트`에서 운영할 수 있다
- 하지만 콘텐츠를 저장할 `DB`와 이미지/파일을 저장할 `스토리지`는 여전히 필요하다

즉, 아래처럼 이해하는 것이 맞다.

| 항목                   | 별도 서버 필요 여부 | 비고                                |
| ---------------------- | ------------------- | ----------------------------------- |
| 공개 사이트            | 불필요              | Vercel에서 실행                     |
| Payload 관리자/CMS API | 불필요              | 같은 Vercel 앱에서 실행 가능        |
| Postgres DB            | 필요                | 서버가 아니라 관리형 DB 서비스 사용 |
| 파일 저장소            | 필요                | Vercel Blob 또는 S3 계열 사용       |

### 실무적으로는 어떻게 보나

`Vercel 단독 운영`이라고 말할 수는 있다. 다만 정확한 의미는 `애플리케이션 서버를 Vercel 하나로 운영`하는 것이다.

비용 청구는 보통 아래처럼 나뉜다.

- Vercel 앱 사용료/함수 사용료
- Postgres 사용료
- Blob 저장소/트래픽 사용료

그래서 `물리 서버 2대 운영`은 아니지만, `앱 + DB + 스토리지`라는 3개 구성요소는 필요하다.

## 6. 비용 관점 평가

### 안 A. Vercel 통합형

장점:

- 서버 관리 인건비를 줄일 수 있다
- 자동 배포, 롤백, 로그, CDN, HTTPS가 기본 제공된다
- `ISR + on-demand revalidation`을 쓰면 모든 요청을 SSR로 처리하지 않아도 된다
- 비용이 실제 사용량 중심이라 초기 트래픽이 작을 때 부담이 덜할 수 있다

주의:

- 함수 호출량이 늘면 사용량 과금이 생긴다
- 이미지/파일 다운로드가 많으면 Blob 비용이 생긴다
- 팀 단위 협업 시 Vercel Pro 비용이 발생할 수 있다

### 안 B. 정적 업로드형 + 별도 CMS

장점:

- 공개 사이트 정적 파일 업로드 비용만 보면 저렴해 보인다

주의:

- Payload는 정적 호스팅만으로 운영할 수 없다
- 결국 별도 Node 실행 환경, DB, 파일 저장소가 필요하다
- 정적 파일 재배포 자동화, CMS 연동, 실시간 반영 설계까지 포함하면 운영비와 개발비가 다시 붙는다
- 담당자가 바뀌거나 운영 기간이 길어질수록 누적 관리비가 커질 가능성이 높다

### 비용 민감한 클라이언트 기준 실무 판단

가장 저렴해 보이는 구조는 `정적 웹호스팅`이지만, 이번 프로젝트 요구사항까지 넣으면 실제 총비용이 꼭 낮아지지 않는다.

오히려 아래 조합이 더 현실적이다.

- 앱: Vercel
- DB: 가장 작은 Postgres 플랜부터 시작
- 파일: 초기에는 최소 용량 Blob 사용
- 렌더링: `정적 + ISR 혼합`

즉, `항상 SSR`이 아니라 `필요한 페이지만 서버/재검증`으로 가져가야 비용이 안정적이다.

## 7. 실시간 반영이 필요한 콘텐츠 정의

실시간 또는 준실시간 반영이 필요한 콘텐츠는 아래 3종이 핵심이다.

| 콘텐츠         | 반영 요구 수준 | 비고                                 |
| -------------- | -------------- | ------------------------------------ |
| 공지사항       | 높음           | 운영 공지, 일정 변경, 모집 공지      |
| 캐스팅/오디션  | 매우 높음      | 마감일, 모집 상태, 진행중 공고       |
| 배우/강사 정보 | 높음           | 프로필, 경력, 대표 이미지, 공개 여부 |

추가로 아래도 재검증 대상에 포함하는 것이 좋다.

- 센터별 목록 페이지
- 통합 목록 페이지
- sitemap.xml

예시:

- 공지사항 상세 수정 시: `/news/[slug]`, `/news`, `/sitemap.xml`
- 캐스팅 상세 수정 시: `/${center}/casting/[slug]`, `/${center}/casting`, `/casting`, `/sitemap.xml`
- 강사 상세 수정 시: `/${center}/faculty/[slug]`, `/${center}/faculty`, `/faculty`, `/sitemap.xml`

## 8. 실시간 반영 구현 방법

### 권장 방식: ISR + on-demand revalidation

이 프로젝트의 기본 전략은 `매 요청 SSR`이 아니라 `정적으로 빠르게 제공하되, 저장 시 필요한 경로만 재생성`하는 방식이 적합하다.

흐름:

1. 관리자가 Payload에서 콘텐츠 저장
2. Payload의 `afterChange` / `afterDelete` 훅 실행
3. 변경된 문서에 연관된 경로를 계산
4. `revalidatePath()` 또는 재검증 API 호출
5. 다음 요청부터 최신 HTML 제공

장점:

- 실시간성에 가깝다
- SSR보다 비용이 낮다
- SEO용 HTML이 유지된다

### 대안 1: SSR

모든 요청마다 DB를 읽어 항상 최신 HTML을 생성하는 방식이다.

장점:

- 저장 즉시 항상 최신
- 구현 개념이 단순하다

단점:

- 비용이 가장 많이 든다
- DB 호출과 함수 실행이 매 요청마다 발생한다
- 트래픽이 늘수록 부담이 커진다

이 프로젝트에서는 `기본 전략`으로는 비권장이다.

### 대안 2: 정적 사이트 + 클라이언트 직접 조회

BNB-Inderstry 프로젝트와 유사하게, 페이지 껍데기는 정적 배포하고 본문 데이터는 브라우저에서 DB를 직접 읽는 방식이다.

장점:

- 게시글 수정은 즉시 보일 수 있다
- 정적 업로드형에서도 동작 가능하다

단점:

- 검색엔진이 처음 받는 HTML에 본문이 비어 있거나 약할 수 있다
- 메타데이터, OG, schema, canonical, 404 제어가 불리하다
- 배우앤배움처럼 SEO가 중요한 상세 페이지에는 부적합하다

이 프로젝트에서는 `운영 우회책`으로만 보고, 기본 구조로는 비추천한다.

## 9. 페이지별 권장 렌더링 전략

| 페이지 유형             | 예시                                    | 권장 방식                    | 이유                  |
| ----------------------- | --------------------------------------- | ---------------------------- | --------------------- |
| 브랜드/정책 고정 페이지 | `/about`, `/privacy`, `/terms`          | 정적 생성                    | 자주 안 바뀜          |
| 센터 소개/가이드        | `/art/about`, `/exam/guide`             | 정적 생성 또는 장주기 ISR    | 운영 빈도 낮음        |

| 캐스팅 목록/상세        | `/art/casting`, `/art/casting/[slug]`   | ISR + on-demand revalidation | 실시간성 중요         |
| 배우/강사 목록/상세     | `/kids/faculty`, `/kids/faculty/[slug]` | ISR + on-demand revalidation | 프로필 변경 반영 필요 |
| 통합 목록 페이지        | `/faculty`, `/casting`                  | ISR + on-demand revalidation | 다수 상세 변경 영향   |
| 관리자 미리보기         | `/preview/...`                          | 동적 렌더링                  | 발행 전 확인용        |

## 10. 안 A 상세 실행 플랜

### 10.1 아키텍처

- Next.js App Router 프로젝트에 공개 사이트와 Payload CMS를 통합
- Vercel에 단일 프로젝트로 배포
- DB는 Vercel Marketplace에서 연결
- 업로드 파일은 Vercel Blob 연결

### 10.2 콘텐츠 반영 정책

- 기본: ISR
- 저장/삭제 시: 관련 경로만 재검증
- 매우 민감한 운영성 콘텐츠는 필요 시 SSR로 승격 가능

### 10.3 운영 절차

1. 콘텐츠 모델 확정
2. Payload 컬렉션 정의
3. 상세 URL 슬러그 전략 확정
4. 페이지별 메타데이터/Schema 연결
5. `afterChange` / `afterDelete` 훅에 재검증 로직 추가
6. sitemap/robots 자동 생성 연결
7. 구 도메인 301 이전 설정
8. Search Console 등록 및 모니터링

### 10.4 재검증 규칙 예시

| 컬렉션      | 변경 시 재검증 경로                                                           |
| ----------- | ----------------------------------------------------------------------------- |
| `news`      | `/news`, `/news/[slug]`, `/sitemap.xml`                                       |
| `faculty`   | `/${center}/faculty`, `/${center}/faculty/[slug]`, `/faculty`, `/sitemap.xml` |
| `casting`   | `/${center}/casting`, `/${center}/casting/[slug]`, `/casting`, `/sitemap.xml` |
| `results`   | `/${center}/results`, `/${center}/results/[slug]`, `/results`, `/sitemap.xml` |
| `centerSeo` | `/${center}`, `/${center}/about`, 관련 섹션 목록                              |

### 10.5 장점 정리

- 의사결정자 설득 포인트가 명확하다
- SEO와 운영성을 동시에 만족한다
- "수정 즉시 반영" 요구를 무리 없이 수용한다
- 비용은 SSR 남발만 피하면 통제 가능하다

## 11. 안 B 상세 실행 플랜

### 11.1 필요한 구성

- 정적 프런트 빌드 및 업로드 파이프라인
- 별도 Payload 실행 서버
- Postgres
- 파일 저장소
- 콘텐츠 변경 시 재배포 자동화 또는 클라이언트 직접 조회 구조

### 11.2 실시간 반영 시나리오

가능한 방법은 2개뿐이다.

- 방법 1. 콘텐츠 저장 시 자동 빌드/배포
- 방법 2. 공개 페이지가 브라우저에서 DB/API를 직접 읽음

### 11.3 문제점

- 방법 1은 즉시성이 떨어지고 실패 지점이 많다
- 방법 2는 SEO 품질이 떨어질 수 있다
- 결국 "정적 웹호스팅이라서 저렴하다"는 장점이 CMS 분리 운영으로 상쇄된다

### 11.4 적용 가능 판단

이 안은 아래 조건에서만 고려할 수 있다.

- SEO보다 초기 고정비가 더 중요할 때
- 공지/캐스팅/배우 상세를 검색 유입용 페이지로 강하게 키우지 않을 때
- 운영팀이 기술적 우회 구조를 감수할 때

배우앤배움에는 적합하지 않다.

## 12. 최종 권장안

### 권장

`안 A. Vercel 통합형` 채택

구체적으로는 아래 조합을 권장한다.

- 앱: Vercel
- CMS: Payload를 같은 Next.js 앱 안에서 운영
- DB: Vercel Marketplace Postgres 또는 동급 관리형 Postgres
- 파일: Vercel Blob
- 렌더링: `정적 + ISR + on-demand revalidation`

### 이유

- 결정권자에게 설명하기 쉽다
- 서버를 여러 대 운영하는 인상이 적다
- 관리자 콘텐츠 수정 반영 요구를 충족한다
- 검색엔진 친화적인 HTML을 유지할 수 있다
- 비용은 `전체 SSR`보다 훨씬 보수적으로 관리할 수 있다

### 비권장

`정적 웹호스팅 업로드형`을 기본안으로 채택하는 것

그 이유는 비용이 아니라 `요구사항 불일치`다.

- Payload CMS와 잘 맞지 않는다
- SEO 목표와 충돌한다
- 콘텐츠 즉시 반영 요구와 충돌한다

## 13. 의사결정자 보고용 한 줄 정리

`Vercel 통합형은 비용이 조금 더 구조화되어 보이지만, 실제로는 SEO/운영/실시간 반영 요구를 가장 낮은 복잡도로 만족하는 안이다. 정적 업로드형은 겉으로만 단순하고, CMS와 실시간 반영 요구를 넣는 순간 오히려 비싸고 복잡해질 가능성이 높다.`
