# 프로젝트 운영 정책

> 목적: 배포, DB, 파일 스토리지, 환경변수 기준을 한 문서로 관리한다.
> 마지막 갱신: 2026-06-30

## 현재 결론

- 앱: `Next.js App Router + Payload CMS`
- 앱 호스팅: `Vercel`
- 로컬 DB: `MariaDB` 원본 복원용, `Postgres` 정제/검증용
- 운영 DB: `Vercel Neon Postgres`
- 파일/이미지 스토리지: `Cloudflare R2`
- 공개 사이트와 Payload admin은 같은 Next.js 앱에서 운영한다.
- 콘텐츠 반영은 정적 페이지, ISR, on-demand revalidation을 조합한다.

## 공개 페이지 캐시와 재검증 정책

목표는 공개 페이지의 응답 속도와 관리자 수정 반영성을 함께 유지하는 것이다. 모든 CMS 연동 페이지를 `force-dynamic`으로 바꾸면 반영은 빠르지만 Vercel 캐시 이점을 잃는다. 따라서 기본값은 `정적 생성 + ISR + 저장 시 on-demand revalidation`이다.

관리자 저장 시에는 기존 캐시를 무효화하고, 다음 공개 요청에서 새 HTML을 생성한 뒤 다시 HIT로 제공하는 흐름을 기준으로 한다. 즉 평소에는 `x-vercel-cache: HIT`를 유지하고, 콘텐츠 변경 시 필요한 경로만 다시 생성되게 한다.

현재 테스트 환경인 Vercel Hobby/free에서도 이 방향을 기본으로 한다. 매 요청 서버 렌더링으로 처리하는 것보다 ISR 캐시를 유지하고 저장 시 필요한 경로만 무효화하는 방식이 무료 티어 사용량에 유리하다. 단, 무료 티어에도 월 단위 사용량 제한은 있으므로 대량 백필이나 수천 건 일괄 저장에서는 `req.context.disableRevalidate` 또는 별도 배치 전략으로 재검증 폭주를 막는다. 라이브 전에는 유료 플랜 전환을 전제로 실제 트래픽, ISR 재검증 빈도, function 사용량을 다시 확인한다.

### 렌더링 방식 선택 기준

| 페이지 유형 | 기본 방식 | 기준 |
| --- | --- | --- |
| 센터 메인, 목록, 아카이브 | `revalidate = 600` + `generateStaticParams()` + 저장 시 `revalidatePath` | SEO와 응답 속도가 중요하고, 관리자 수정은 저장 직후 반영되어야 한다. |
| 상세 페이지 | 현재 문서 상태에 따라 `force-dynamic` 또는 ISR | 상세 수정이 자주 일어나거나 미리보기/비공개 확인이 중요하면 `force-dynamic`을 허용한다. 조회량이 큰 상세는 ISR 전환을 검토한다. |
| 관리자 확인, 테스트, 일정처럼 요청마다 상태가 달라지는 페이지 | `force-dynamic` | 사용자 입력, 현재 날짜, 권한, query string이 결과를 크게 바꾸면 캐시하지 않는다. |
| 완전 정적 안내 페이지 | 정적 생성 | CMS 데이터 의존이 없고 수정 빈도가 낮으면 재검증 hook을 만들지 않는다. |

### 대량 상세 페이지 사전 생성 기준

페이지 수가 많은 공개 컬렉션의 상세 페이지는 전체 문서를 빌드 시점에 모두 사전 생성하지 않는다. 기본값은 `generateStaticParams()`에서 빈 배열을 반환하고, URL은 `dynamicParams = true`로 열어 둔다.

단, 상세 렌더링 경로에서 `draftMode()`, request cookie/header, 권한 판별처럼 Next 동적 API를 사용하면 `dynamic = "force-dynamic"`을 명시하고 `generateStaticParams()`를 export하지 않는다. 이런 라우트는 정적 ISR로 첫 요청 생성하지 않고 서버에서 동적으로 렌더링한다. 동적 API 오류를 `.catch(() => null)`로 삼키면 Vercel에서 `DYNAMIC_SERVER_USAGE` 500이 날 수 있으므로, 동적 API를 쓰는 상세는 정적 라우트로 남기지 않는다.

기본 기준:

- 대상: `news`, `artist-press`, `screen-appearances`, `exam-results`, `profiles`, `teachers`, `curriculums`처럼 상세 URL이 많아질 수 있는 공개 컬렉션 전체
- 사전 생성 범위: 기본 0개. 이용자 체감이 큰 `news`, `exam-results`, `exam-passed-reviews` 같은 일부 상세만 섹션 또는 공개 목록 단위별 최신 10개 안팎으로 제한한다.
- 정렬 기준: 가능하면 `publishedAt`/`createdAt`/관리자 정렬값 중 공개 목록과 같은 최신순 기준
- 캐시 기준: 테스트 단계에서는 정적 ISR 상세의 `revalidate = 600`을 유지한다. `force-dynamic` 상세는 이 값보다 서버 동적 렌더링 설정이 우선한다. 실제 운영 도메인 전환 후 변경 빈도와 트래픽을 보고 더 긴 값 또는 별도 공개 캐시 전략으로 조정한다.
- 오래된 상세: 빌드에서 제외하되 URL을 막지 않는다. 정적 ISR 가능한 상세는 `dynamicParams = true`로 첫 접근 시 생성하고 이후 ISR 캐시로 제공한다. `draftMode()` 등 동적 API가 필요한 상세는 `force-dynamic`으로 정상 응답을 우선한다.
- SEO 기준: 사전 생성 대상이 아니어도 정상 상세 URL, metadata, canonical, sitemap 정책은 유지한다. 사전 생성 제한은 빌드 비용 조절 정책이지 색인 제외 정책이 아니다.

새 공개 상세 라우트를 만들거나 기존 `generateStaticParams()`를 수정할 때는 위 기준을 먼저 적용한다. 전체 문서 사전 생성 또는 섹션별 100개 이상 사전 생성이 꼭 필요하면 데이터 건수, Vercel 빌드 시간, 원격 DB 부하 근거를 남긴 뒤 예외로 처리한다.

`x-vercel-cache: HIT`는 Vercel이 캐시된 응답을 제공했다는 의미다. 이 상태 자체는 문제가 아니다. 문제는 해당 페이지가 읽는 Payload 데이터가 바뀌었는데 관련 공개 경로를 재검증하지 않아 오래된 HTML이 계속 노출되는 경우다.

### 재검증 소유권

Payload 컬렉션이나 글로벌 설정이 공개 페이지에 노출되면, 그 데이터를 저장하는 컬렉션/글로벌 설정 hook이 관련 공개 경로를 재검증할 책임을 가진다.

원칙:

- `afterChange`에서 관련 공개 경로를 `revalidatePath(path, "page")`로 재검증한다.
- 삭제가 공개 목록이나 홈 섹션에서 사라져야 하는 데이터는 `afterDelete`도 같이 처리한다.
- `req.context.disableRevalidate`가 있으면 재검증을 건너뛴다. 테스트, 마이그레이션, 대량 백필에서 의도적으로 재검증을 끌 수 있어야 한다.
- 센터 필드가 있는 문서는 현재 센터와 이전 센터를 모두 재검증한다. 센터를 옮겼을 때 이전 목록에 남는 문제를 막기 위해서다.
- `centers: all` 문서는 모든 센터 경로를 재검증한다.
- 페이지가 목록 API를 추가로 호출하면 HTML 경로와 API 응답 정책을 함께 본다. API가 캐시되지 않아도 초기 HTML이 오래되면 사용자는 오래된 목록을 볼 수 있다.
- `unstable_cache`를 쓰는 데이터 함수는 가능하면 tag 또는 path 재검증 계획을 같이 둔다. 경로 재검증만으로 충분하지 않은 구조라면 `revalidateTag`를 병행한다.

### 센터 메인 재검증 기준

센터 메인 `/{center}`는 여러 컬렉션의 요약 데이터를 함께 보여주므로 가장 보수적으로 관리한다. 센터 메인에 연결되는 데이터가 저장되면 해당 센터 메인도 재검증 대상에 포함한다.

현재 센터 메인 의존 데이터:

| 데이터 | 현재 공개 영향 | 재검증 대상 |
| --- | --- | --- |
| `main` 글로벌 | 센터별 메인 배너 순서와 배너 설정 | 모든 `/{center}` |
| `main-banners` | 센터 메인 히어로 배너 | 현재 센터와 이전 센터의 `/{center}` |
| `main-statistics` | 메인 통계 영역 | 모든 `/{center}` |
| `screen-appearances` | 센터 메인 출연장면 요약, 출연장면 목록/상세 | 해당 센터 `/{center}`, `/{center}/screen-appearances` |
| `artist-press` | 센터 메인 아티스트 소식 요약, 아티스트 소식 목록/상세 | 해당 센터 `/{center}`, `/{center}/artist-press` |
| `news` | 센터 메인 뉴스 요약, 뉴스 목록/상세 | 해당 센터 `/{center}`, `/{center}/news` |
| `social-links` | 센터 메인 SNS 섹션 | 해당 센터 `/{center}` |
| `curriculums` | 센터 메인 수업 시간 요약, 커리큘럼 목록/상세 | 해당 센터 `/{center}`, `/{center}/curriculum` |
| `exam-passed-reviews` | 입시센터 메인 합격 후기 요약, 합격 후기 목록/상세 | `/exam`, `/exam/passed-reviews` |
| `exam-passed-videos` | 입시센터 메인 합격 영상 요약, 합격 영상 목록 | `/exam`, `/exam/passed-videos` |
| `footer` 글로벌 | 센터 메인 SNS 계정 링크와 공통 푸터 | 모든 공개 페이지 또는 최소 모든 `/{center}` |

위 표에 새 컬렉션을 추가하거나 홈 섹션의 데이터 소스를 바꾸면, 같은 변경에서 재검증 hook 또는 재검증 정책을 같이 수정한다. 표시 컴포넌트만 바꾸고 재검증 경로를 빼먹으면 미완료로 본다.

### 목록 페이지 재검증 기준

목록 페이지는 변경 빈도가 낮아도 관리자 수정 후 바로 반영되어야 한다. `revalidate = 600`은 장애 시 자동 회복을 위한 보조 안전망이고, 운영 반영 기준은 저장 시 재검증이다.

현재 기준:

| 컬렉션 | 공개 목록 | 저장 시 재검증 기준 |
| --- | --- | --- |
| `casting-appearances` | `/{center}/casting-status` | 반영됨. 현재 센터, 이전 센터, `all` 센터 확장 |
| `screen-appearances` | `/{center}/screen-appearances` | 반영됨. 센터 메인 `/{center}`도 함께 재검증 |
| `artist-press` | `/{center}/artist-press` | 반영됨. 센터 메인 `/{center}`도 함께 재검증 |
| `news` | `/{center}/news` | 반영됨. 센터 메인 `/{center}`도 함께 재검증 |
| `direct-castings` | `/{center}/direct-castings` | 반영됨. 현재 센터와 이전 센터를 재검증 |
| `teachers` | `/{center}/teachers` | 반영됨. 센터 메인 `/{center}`도 함께 재검증 |
| `curriculums` | `/{center}/curriculum` | 반영됨. 센터 메인 `/{center}`도 함께 재검증 |

새 작업에서 위 컬렉션의 공개 경로가 늘어나거나 새 컬렉션을 공개 페이지에 연결하면 재검증 hook과 이 표를 함께 갱신한다.

### 구현 체크리스트

CMS 데이터를 공개 페이지에 새로 연결하거나 기존 컬렉션의 공개 노출 방식을 바꿀 때는 아래를 확인한다.

1. 공개 경로를 모두 적는다. 예: `/{center}`, `/{center}/news`, `/{center}/news/{slug}`.
2. 페이지 렌더링 방식이 `force-dynamic`, ISR, 완전 정적 중 무엇인지 확인한다.
3. ISR 또는 정적 생성이면 저장/삭제 hook에서 `revalidatePath` 또는 `revalidateTag`를 호출한다.
4. 센터 변경, 공개/비공개 전환, 삭제, 발행일 변경이 목록 순서와 홈 요약에 미치는 영향을 확인한다.
5. 테스트에서는 경로 계산 함수를 분리해 현재 센터, 이전 센터, `all` 확장을 검증한다.
6. 배포 후 문제가 의심되면 `curl -I`로 `x-vercel-cache`, `x-nextjs-prerender`, `cache-control`을 확인하고, HTML 안의 실제 데이터 문자열까지 확인한다.

## R2 이전 정책

현재 개발은 개발자 Cloudflare 계정의 R2에서 먼저 테스트한다. 운영 전에는 클라이언트 Cloudflare 계정의 R2로 객체를 복사한다.

이전이 쉬워야 하므로 DB/Payload/본문에는 개발자 계정에 종속된 완성 URL을 저장하지 않는다.

저장 기준:

- 정본은 `objectKey`다.
- 공개 URL은 런타임에서 `R2_PUBLIC_BASE_URL + objectKey`로 만든다.
- 개발자 R2와 클라이언트 R2는 같은 `objectKey` 구조를 유지한다.
- 계정 이전 시 객체를 같은 key로 복사하고 환경변수만 교체한다.
- as-is 사이트가 사용 중인 도메인은 R2에 나중에 연결한다. 연결 전에는 임시 `r2.dev` 공개 URL을 `R2_PUBLIC_BASE_URL`로 쓰고, 연결 후에는 같은 `objectKey`를 유지한 채 `R2_PUBLIC_BASE_URL`만 커스텀 도메인으로 교체한다.

저장 금지:

- `https://...r2.dev/...` 전체 URL
- 개발자 임시 커스텀 도메인 전체 URL
- 특정 Cloudflare `account_id`가 들어간 endpoint
- `R2_PUBLIC_BASE_URL`이 포함된 완성 public URL

권장 key 구조:

```text
legacy/{collection}/{source_db}/{bo_table}/{source_id}/{asset_role}/{filename}
media/{collection}/{asset_role}/{source_id}/{filename}
```

R2 object key 작성 규칙:

- R2 콘솔에서 폴더처럼 탐색할 수 있도록 최종 object key에는 반드시 `/` 경로 구분을 둔다.
- Payload media를 생성할 때도 `prefix`와 `filename`을 합친 최종 key가 `media/{collection}/{role}/{source_id}/{filename}`처럼 계층형인지 확인한다.
- `direct-castings-body-11-874bfd79b6-...jpg`처럼 컬렉션/역할/출처를 하이픈 파일명 하나에 몰아넣은 평탄한 key는 금지한다.
- R2에 새로 올리는 정제 파일명은 짧게 유지한다. 기본은 `{role-base}-{id}.{ext}`이고 Payload size 파일은 `{role-base}-{id}-{size}.{ext}`를 쓴다. 예: `exam-school-logo-1.png`, `casting-director-10-thumbnail.png`.
- Lexical 본문 이미지처럼 본문 안에서 원본 이미지만 렌더링하는 역할은 size 파생본을 R2에 올리지 않는다. `media/{collection}/body-images/{source_id}/{role-base}-{source_id}-{index}.{ext}` 형태로 원본만 저장하고, Payload `sizes_*_filename` 값은 비워 존재하지 않는 size URL이 내려가지 않게 한다.
- media 업로드/백필 스크립트는 dry-run 또는 write 리포트에 최종 object key 샘플을 남겨서 R2 경로 구조를 검증할 수 있게 한다.
- 새 대량 업로드는 `src/lib/r2ObjectKeys.ts`의 `R2_MEDIA_PREFIX_BY_ROLE` allowlist를 기준으로 한다. 새 컬렉션/역할을 추가할 때는 먼저 이 allowlist에 `collection.asset-role -> media/{collection}/{asset_role}` 매핑을 추가한다.
- 기존에 이미 동작 중인 평탄 key는 강제 재작성하지 않는다. 정리가 필요하면 R2 복사와 DB/Payload 참조 변경을 별도 마이그레이션으로 분리한다.

```text
R2_ACCESS_KEY_ID=
R2_SECRET_ACCESS_KEY=
R2_BUCKET=
R2_ENDPOINT=
R2_PUBLIC_BASE_URL=
```

`R2_ACCOUNT_ID`는 Cloudflare 계정 식별과 운영 기록용으로 남길 수 있지만, 현재 앱 런타임의 R2 client는 직접 사용하지 않는다.

클라이언트 계정 이전 시 `objectKey`는 유지하고 위 R2 환경변수와 공개 도메인만 교체한다. Payload + R2 + Vercel 세부 기준은 [03-Payload-R2-Vercel-운영-주의사항.md](./03-Payload-R2-Vercel-운영-주의사항.md)를 따른다.

## 로컬 개발 기준

```bash
npm install
docker compose up -d
npm run db:migrate
npm run dev
```

레거시 MariaDB dump를 함께 사용할 때:

```bash
npm run legacy:db:up
npm run legacy:db:import
npm run legacy:db:verify
```

현재 데이터 이전 흐름:

```text
data/legacy_dumps
  -> 로컬 MariaDB
  -> 로컬 Postgres/Payload DB 정제
  -> Vercel Neon
```

과거 P0/P1/P2/C0 기준은 현재 이관 판단에 사용하지 않는다.

## DB 구분 기록 기준

문서, 체크리스트, 검증 로그에 DB 관련 내용을 남길 때는 대상 DB를 생략하지 않는다.

기준 명칭:

- `로컬 MariaDB`: 레거시 원본 dump 복원과 `bnb_legacy_work.*` work table 확인 대상
- `로컬 Postgres/Payload DB`: Payload 컬렉션 기준 정제 적재와 로컬 검증 대상
- `원격 Vercel Neon DB`: 로컬 검증 완료 후 schema/data 이전 대상

금지 표현:

- `DB 반영 완료`
- `테이블 확인 완료`
- `마이그레이션 완료`

위 표현은 어느 DB를 의미하는지 모호하므로 `로컬 Postgres/Payload DB migration 적용 완료`, `로컬 MariaDB work table 카운트 확인`, `원격 Vercel Neon DB 이전 완료`처럼 대상과 작업을 함께 적는다.

## 원격 Payload 스키마 마이그레이션

신규 컬렉션, 필드, 글로벌 스키마 변경을 원격 Vercel Neon DB에 반영할 때는 로컬 검증이 끝난 migration 파일만 적용한다.

적용 전 확인:

- `.env.local`의 원격 DB URL이 `원격 Vercel Neon DB`를 가리키는지 확인한다.
- 생성된 `src/migrations/*.ts`와 `src/migrations/index.ts`가 커밋 대상에 포함되어 있는지 확인한다.
- 로컬 Postgres/Payload DB에서 `npm run db:local:migrate`, `npm run payload:generate-types`, `npm run typecheck`를 먼저 통과시킨다.

원격 적용 명령:

```bash
npm run db:remote:migrate
```

적용 후 확인:

- Payload admin에서 신규 컬렉션 메뉴가 보이는지 확인한다.
- 원격 DB에 신규 테이블이 생성됐는지 확인한다.
- 공개 프론트에서 해당 데이터 조회가 필요한 경우 실제 route 응답을 확인한다.

## 검증 기준

기능 변경 후 가능한 범위에서 아래를 실행한다.

```bash
npm run payload:generate-types
npm run payload:generate-importmap
npm run typecheck
npm run lint
npm run build
```

`npm run build`가 `next-env.d.ts`의 route type 경로를 `.next/types/routes.d.ts`로 바꾸면 기존 dev 경로인 `.next/dev/types/routes.d.ts`로 되돌려 diff를 깨끗하게 유지한다.
