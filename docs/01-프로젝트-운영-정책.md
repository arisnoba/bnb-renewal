# 프로젝트 운영 정책

> 목적: 배포, DB, 파일 스토리지, 환경변수 기준을 한 문서로 관리한다.
> 마지막 갱신: 2026-04-28

## 현재 결론

- 앱: `Next.js App Router + Payload CMS`
- 앱 호스팅: `Vercel`
- 로컬 DB: `MariaDB` 원본 복원용, `Postgres` 정제/검증용
- 운영 DB: `Vercel Neon Postgres`
- 파일/이미지 스토리지: `Cloudflare R2`
- 공개 사이트와 Payload admin은 같은 Next.js 앱에서 운영한다.
- 콘텐츠 반영은 정적 페이지, ISR, on-demand revalidation을 조합한다.

## R2 이전 정책

현재 개발은 개발자 Cloudflare 계정의 R2에서 먼저 테스트한다. 운영 전에는 클라이언트 Cloudflare 계정의 R2로 객체를 복사한다.

이전이 쉬워야 하므로 DB/Payload/본문에는 개발자 계정에 종속된 완성 URL을 저장하지 않는다.

저장 기준:

- 정본은 `objectKey`다.
- 공개 URL은 런타임에서 `R2_PUBLIC_BASE_URL + objectKey`로 만든다.
- 개발자 R2와 클라이언트 R2는 같은 `objectKey` 구조를 유지한다.
- 계정 이전 시 객체를 같은 key로 복사하고 환경변수만 교체한다.
- as-is 사이트가 사용 중인 도메인은 R2에 나중에 연결한다. 연결 전에는 임시 `r2.dev` 공개 URL을 `R2_PUBLIC_BASE_URL`로 쓰고, 연결 후에는 같은 `objectKey`를 유지한 채 `R2_PUBLIC_BASE_URL`만 커스텀 도메인으로 교체한다.

저장 금지:

- `https://...r2.dev/...` 전체 URL
- 개발자 임시 커스텀 도메인 전체 URL
- 특정 Cloudflare `account_id`가 들어간 endpoint
- `R2_PUBLIC_BASE_URL`이 포함된 완성 public URL

권장 key 구조:

```text
legacy/{collection}/{source_db}/{bo_table}/{source_id}/{asset_role}/{filename}
media/{collection}/{asset_role}/{source_id}/{filename}
```

R2 object key 작성 규칙:

- R2 콘솔에서 폴더처럼 탐색할 수 있도록 최종 object key에는 반드시 `/` 경로 구분을 둔다.
- Payload media를 생성할 때도 `prefix`와 `filename`을 합친 최종 key가 `media/{collection}/{role}/{source_id}/{filename}`처럼 계층형인지 확인한다.
- `direct-castings-body-11-874bfd79b6-...jpg`처럼 컬렉션/역할/출처를 하이픈 파일명 하나에 몰아넣은 평탄한 key는 금지한다.
- R2에 새로 올리는 정제 파일명은 짧게 유지한다. 기본은 `{role-base}-{id}.{ext}`이고 Payload size 파일은 `{role-base}-{id}-{size}.{ext}`를 쓴다. 예: `exam-school-logo-1.png`, `casting-director-10-thumbnail.png`.
- media 업로드/백필 스크립트는 dry-run 또는 write 리포트에 최종 object key 샘플을 남겨서 R2 경로 구조를 검증할 수 있게 한다.
- 새 대량 업로드는 `src/lib/r2ObjectKeys.ts`의 `R2_MEDIA_PREFIX_BY_ROLE` allowlist를 기준으로 한다. 새 컬렉션/역할을 추가할 때는 먼저 이 allowlist에 `collection.asset-role -> media/{collection}/{asset_role}` 매핑을 추가한다.
- 기존에 이미 동작 중인 평탄 key는 강제 재작성하지 않는다. 정리가 필요하면 R2 복사와 DB/Payload 참조 변경을 별도 마이그레이션으로 분리한다.

## 필수 환경변수

앱 기본값:

```text
DATABASE_URL=
PAYLOAD_SECRET=
NEXT_PUBLIC_SITE_URL=
```

R2:

```text
R2_ACCOUNT_ID=
R2_ACCESS_KEY_ID=
R2_SECRET_ACCESS_KEY=
R2_BUCKET=
R2_ENDPOINT=
R2_PUBLIC_BASE_URL=
```

클라이언트 계정 이전 시 `objectKey`는 유지하고 위 R2 환경변수와 공개 도메인만 교체한다.

## 로컬 개발 기준

```bash
npm install
docker compose up -d
npm run db:migrate
npm run dev
```

레거시 MariaDB dump를 함께 사용할 때:

```bash
npm run legacy:db:up
npm run legacy:db:import
npm run legacy:db:verify
```

현재 데이터 이전 흐름:

```text
data/legacy_dumps
  -> 로컬 MariaDB
  -> 로컬 Postgres/Payload DB 정제
  -> Vercel Neon
```

과거 P0/P1/P2/C0 기준은 현재 이관 판단에 사용하지 않는다.

## DB 구분 기록 기준

문서, 체크리스트, 검증 로그에 DB 관련 내용을 남길 때는 대상 DB를 생략하지 않는다.

기준 명칭:

- `로컬 MariaDB`: 레거시 원본 dump 복원과 `bnb_legacy_work.*` work table 확인 대상
- `로컬 Postgres/Payload DB`: Payload 컬렉션 기준 정제 적재와 로컬 검증 대상
- `원격 Vercel Neon DB`: 로컬 검증 완료 후 schema/data 이전 대상

금지 표현:

- `DB 반영 완료`
- `테이블 확인 완료`
- `마이그레이션 완료`

위 표현은 어느 DB를 의미하는지 모호하므로 `로컬 Postgres/Payload DB migration 적용 완료`, `로컬 MariaDB work table 카운트 확인`, `원격 Vercel Neon DB 이전 완료`처럼 대상과 작업을 함께 적는다.

## 검증 기준

기능 변경 후 가능한 범위에서 아래를 실행한다.

```bash
npm run payload:generate-types
npm run payload:generate-importmap
npm run typecheck
npm run lint
npm run build
```

`npm run build`가 `next-env.d.ts`의 route type 경로를 `.next/types/routes.d.ts`로 바꾸면 기존 dev 경로인 `.next/dev/types/routes.d.ts`로 되돌려 diff를 깨끗하게 유지한다.
