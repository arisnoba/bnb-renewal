# BNB Renewal 프로젝트 작업 규칙

> 작성일: 2026-04-13
> 목적: 프로젝트 범위, 구현 방식, 검증 기준, 문서 운영 원칙을 하나의 기준으로 맞춘다.

이 문서는 이 저장소에서 의사결정과 구현 방식을 맞추기 위한 기준 문서다.

목표는 단순하다.

- 프로젝트 범위를 불필요하게 키우지 않는다.
- 레거시 이관 리스크를 통제한다.
- 누구나 이어서 작업할 수 있는 상태를 유지한다.

## 1. 프로젝트 목표

이 프로젝트는 기존 배우앤배움 5개 사이트를 하나의 통합 서비스로 재구성하는 작업이다.

핵심 목표는 아래와 같다.

- 센터별 사이트를 하나의 코드베이스로 통합한다.
- 공개 사이트와 관리자 CMS를 같은 제품 흐름 안에서 운영한다.
- 레거시 그누보드 데이터를 현재 확정된 DB 이전 흐름에 맞게 정제 이관한다.
- SEO와 운영 효율을 동시에 유지한다.

## 2. 의사결정 원칙

### 2-1. 기본 원칙

- 정답성은 영리함보다 우선한다.
- 가장 작은 변경으로 해결한다.
- 기존 패턴과 현재 문서 기준을 먼저 따른다.
- 새 추상화, 새 의존성, 대규모 리팩터링은 마지막 수단으로 본다.
- 추측으로 마무리하지 않는다. 가능하면 실행 결과로 확인한다.

### 2-2. 범위 원칙

- 현재 작업에 필요한 범위만 수정한다.
- 인접 코드가 조금 불편해 보여도 이번 작업과 직접 관련이 없으면 건드리지 않는다.
- 과거 P0/P1/P2/C0 기준은 현재 작업 판단에 사용하지 않는다.
- 현재 기준은 `data/legacy_dumps -> 로컬 MariaDB -> 로컬 Postgres 정제 -> Vercel Neon` 흐름이다.
- 개인정보가 포함된 데이터는 기능 편의보다 보존 필요성과 운영 정책을 먼저 검토한다.

### 2-3. 문서 우선 원칙

- 구현 전에 관련 문서를 먼저 읽는다.
- 새 의사결정이 생기면 코드만 바꾸지 말고 기준 문서도 함께 갱신한다.
- 이미 같은 주제를 다루는 문서가 있으면 새 문서를 추가하기보다 기존 문서를 보강한다.

## 3. 작업 절차

비사소한 작업은 아래 순서로 진행한다.

1. 관련 코드와 기존 문서를 읽고 현재 상태를 파악한다.
2. 이번 작업의 성공 조건과 비범위를 짧게 정리한다.
3. 작은 단위로 구현한다.
4. 가능한 검증을 수행한다.
5. 결과와 남은 위험을 기록한다.

다음 상황에서는 기능 추가를 멈추고 먼저 재정리한다.

- 예상보다 수정 범위가 빠르게 커질 때
- 기존 문서와 실제 구현이 크게 어긋날 때
- 데이터 손실, SEO 저하, 운영 회귀 가능성이 보일 때

## 4. 코드 변경 규칙

### 4-1. 구현 방식

- `src/app`, `src/collections`, `src/lib`의 기존 구조를 우선 따른다.
- 센터 구분은 이미 정의된 slug 체계 `art`, `avenue`, `exam`, `highteen`, `kids`를 기준으로 맞춘다.
- 컬렉션 설계는 "레거시 테이블 복제"가 아니라 "새 서비스 개념 모델" 기준으로 유지한다.
- 디버그 출력은 문제 해결 후 제거한다.
- 장기 가치가 없는 임시 코드, 임시 플래그, 임시 데이터는 남기지 않는다.

### 4-2. 데이터/마이그레이션

- `data/legacy_dumps`를 로컬 MariaDB에 복원한 뒤 work table을 구성한다.
- 정제된 운영 데이터는 로컬 Postgres/Payload DB에서 먼저 만든다.
- 로컬 검증 후 Vercel Neon으로 migration/데이터 이전을 진행한다.
- 원본 테이블 구조를 1:1 재현하려 하지 않는다.
- 대량 write와 운영 DB 반영은 dry-run, 샘플 검증, 명시적 승인 뒤에 진행한다.

### 4-3. 라우트/SEO

- URL은 통합 IA와 SEO 문서 기준을 우선한다.
- slug 규칙은 안정성과 예측 가능성을 우선한다.
- 기존 도메인에서 새 경로로의 이전을 고려해 경로 변경은 신중하게 처리한다.

## 5. 문서 관리 규칙

### 5-1. 문서 위치

- 루트 `README.md`: 프로젝트 전체 개요와 진입점
- `docs/00-문서-인덱스.md`: 현재 문서 진입점과 archive 맵
- `docs/01~05`: 현재 기준 정책 문서
- `docs/archive`: 과거 상세 기록과 판단 근거
- `plan/현재작업.md`: 현재 진행 상황과 검증 기록
- `plan/archive`: 과거 계획 문서
- `deliverables/`: 외부 제출 또는 산출 결과물

### 5-2. 문서 작성 방식

- 문서는 "왜 이 결정을 했는가"가 드러나야 한다.
- 요약을 먼저 쓰고, 세부 근거는 뒤에 적는다.
- 날짜와 목적을 상단에 남긴다.
- 아직 확정되지 않은 내용은 확정처럼 쓰지 않는다.
- 후속 작업이 남으면 문서에 다음 단계까지 적는다.

## 6. 검증 규칙

완료 처리 전에 가능한 검증을 먼저 수행한다.

기본 검증 후보:

- `npm run lint`
- `npm run typecheck`
- `npm run build`
- `npm run db:migrate`
- `npm run legacy:db:verify`

검증 결과는 아래 원칙으로 공유한다.

- 실제로 실행한 것만 "실행함"이라고 적는다.
- 실행하지 못한 검증은 생략하지 말고 이유를 적는다.
- 데이터 이관 작업은 샘플 또는 dry-run 결과를 우선 확인한다.

## 7. 커뮤니케이션 규칙

- 결과와 영향부터 짧게 설명한다.
- 파일 경로, 명령어, 에러 메시지처럼 확인 가능한 근거를 우선 제시한다.
- 막히지 않았다면 질문을 최소화한다.
- 질문이 꼭 필요하면 한 번에 하나만 묻고, 권장 기본안을 함께 둔다.
- 가정한 내용과 검증 한계를 분리해서 적는다.

## 8. Git 규칙

- 커밋 메시지는 한국어를 기본으로 쓴다.
- 영어 Conventional Commit 접두사는 사용할 수 있다.
- 접두사 뒤 본문은 한국어만 사용한다.

허용 예시:

- `feat: 강사진 컬렉션 필드 구조 정리`
- `fix: 센터별 라우트 slug 검증 오류 수정`
- `docs: 프로젝트 개요와 작업 규칙 문서 추가`

## 9. 완료 기준

작업을 끝냈다고 보려면 아래 질문에 대부분 `예`라고 답할 수 있어야 한다.

- 변경 이유가 문서나 코드에서 바로 보이는가
- 수정 범위가 현재 작업 목적에 비해 과하지 않은가
- 기존 패턴을 불필요하게 깨지 않았는가
- 최소 한 가지 이상 검증했는가
- 못 한 검증과 남은 위험을 명확히 남겼는가

이 기준을 만족하지 못하면 "완료"보다 "중간 상태"로 취급하는 것이 맞다.
