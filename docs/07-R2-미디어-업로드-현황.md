# R2 미디어 업로드 현황

> 목적: Payload `media`와 Cloudflare R2 업로드 진행 상황을 한눈에 확인한다.
> 마지막 확인: 2026-05-20

## 현재 기준

- DB/Payload/본문에는 `R2_PUBLIC_BASE_URL`이 포함된 완성 URL을 저장하지 않는다.
- R2 object key는 `media/{collection}/{asset-role}/{source_id}/{filename}` 계층을 사용한다.
- 정제 후 파일명은 짧게 만든다.
- 본문 이미지(`*.body-image`)는 현재 Lexical `body` upload node가 실제 참조하는 원본만 R2에 올린다.
- 본문 이미지의 Payload size 파생 파일명은 `NULL`로 비운다. 존재하지 않는 `thumbnail`, `square`, `og` URL이 API로 내려가지 않게 하기 위해서다.
- 썸네일, 로고, 프로필 이미지처럼 Payload size가 실제로 쓰일 수 있는 단일 대표 이미지는 기존 size 파일도 함께 올릴 수 있다.

## 다이렉트캐스팅 상세

썸네일:

```text
media/direct-castings/thumbnails/3/direct-casting-thumbnail-3.png
media/direct-castings/thumbnails/3/direct-casting-thumbnail-3-thumbnail.png
media/direct-castings/thumbnails/3/direct-casting-thumbnail-3-square.png
```

본문 이미지:

```text
media/direct-castings/body-images/3/direct-casting-body-image-3-1.png
media/direct-castings/body-images/3/direct-casting-body-image-3-2.png
```

본문 이미지는 처음에 size 파생본까지 partial 업로드된 객체 2,085개를 삭제한 뒤, 원본 698개만 다시 업로드했다.

현재 로컬 Postgres/Payload DB 확인 결과:

- `media.prefix like 'media/direct-castings/body-images/%'`: 698 rows
- 본문 이미지 row 중 `sizes_*_filename`이 남은 row: 0
- R2 `media/direct-castings/body-images/`: 698 objects
- 공개 URL 샘플: 200 응답 확인

## 업로드 전 체크

- 로컬 Postgres/Payload DB 대상인지 확인한다.
- R2 대상 prefix에 기존 partial/stale object가 있는지 확인한다.
- `media` row 수와 실제 콘텐츠가 참조하는 media id 수를 분리해서 본다.
- 본문 이미지는 현재 Lexical `body` upload node가 참조하는 media만 업로드한다.
- 본문 이미지 업로드 후에는 `sizes_thumbnail_filename`, `sizes_square_filename`, `sizes_small_filename`, `sizes_medium_filename`, `sizes_large_filename`, `sizes_xlarge_filename`, `sizes_og_filename`이 모두 `NULL`인지 확인한다.
- 공개 URL 샘플은 `HEAD` 또는 `GET`으로 200 응답과 content-type을 확인한다.

## 검증에 사용한 기준

- R2 count는 Cloudflare R2 S3 `ListObjectsV2`로 prefix별 확인했다.
- 로컬 DB count는 `postgresql://postgres:postgres@127.0.0.1:5432/bnb_renewal` 기준으로 확인했다.
- R2 공개 URL은 현재 `R2_PUBLIC_BASE_URL` 기준으로 확인했다.
