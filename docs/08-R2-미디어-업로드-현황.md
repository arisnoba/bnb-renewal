# R2 미디어 업로드 현황

> 목적: Payload `media`와 Cloudflare R2 업로드 진행 상황을 한눈에 확인한다.
> 마지막 확인: 2026-06-25

## 현재 기준

- DB/Payload/본문에는 `R2_PUBLIC_BASE_URL`이 포함된 완성 URL을 저장하지 않는다.
- R2 object key는 `media/{collection}/{asset-role}/{content_id}/{filename}` 계층을 사용한다.
- 정제 후 파일명은 짧게 만든다.
- 본문 이미지(`*.body-image`)는 현재 Lexical `body` upload node가 실제 참조하는 원본만 R2에 올린다.
- 본문 이미지의 Payload size 파생 파일명은 `NULL`로 비운다. 존재하지 않는 `thumbnail`, `square`, `og` URL이 API로 내려가지 않게 하기 위해서다.
- 썸네일, 로고, 프로필 이미지처럼 단일 대표 이미지도 원본 1개만 업로드하고 Payload size 파생 파일명은 비운다.
- 신규 R2 대상은 `src/lib/r2ObjectKeys.ts`의 `R2_MEDIA_PREFIX_BY_ROLE` allowlist 기준으로 점검한다.

## 다음 작업 권장 순서

1. `main_banners`, `social_links`, `classrooms_rels`의 `media/assets/{mediaId}` 참조를 role별 prefix로 정규화할지 결정한다.
2. 전체 `media` row 기준 R2 누락처럼 보이는 401건은 운영 참조가 없는 stale/orphan 후보로 따로 정리한다.

## 업로드 전 체크

- 로컬 Postgres/Payload DB 대상인지 확인한다.
- R2 대상 prefix에 기존 partial/stale object가 있는지 확인한다.
- `media` row 수와 실제 콘텐츠가 참조하는 media id 수를 분리해서 본다.
- media 업로드는 기본적으로 모든 원본을 품질 80 기준으로 압축하고 파생본은 올리지 않는다. 압축을 끄려면 `--no-compress`, 품질을 바꾸려면 `--compression-quality <1-100>`을 사용한다.
- `news.thumbnail`, `news.body-image`, `star-cards.image`는 `payload:legacy:upload-*:r2` 명령으로 동일한 정책을 적용한다.
- 본문 이미지는 현재 Lexical `body` upload node가 참조하는 media만 업로드한다.
- 본문 이미지 업로드 후에는 `sizes_thumbnail_filename`, `sizes_square_filename`, `sizes_small_filename`, `sizes_medium_filename`, `sizes_large_filename`, `sizes_xlarge_filename`, `sizes_og_filename`이 모두 `NULL`인지 확인한다.
- 공개 URL 샘플은 `HEAD` 또는 `GET`으로 200 응답과 content-type을 확인한다.

대상별 실행 명령:

```bash
npm run payload:legacy:upload-news-thumbnails:r2 -- --dry-run
npm run payload:legacy:upload-news-thumbnails:r2 -- --write

npm run payload:legacy:upload-news-body-images:r2 -- --dry-run
npm run payload:legacy:upload-news-body-images:r2 -- --write

npm run payload:legacy:upload-star-card-images:r2 -- --dry-run
npm run payload:legacy:upload-star-card-images:r2 -- --write
```

## 검증에 사용한 기준

- R2 count는 Cloudflare R2 S3 `ListObjectsV2`로 prefix별 확인했다.
- 2026-06-25 신규 누락 항목은 `.env.local`의 Neon `neondb` 기준으로 확인했다.
- 기존 완료 항목은 업로드 당시 검증값과 현재 R2 prefix count를 함께 기준으로 본다.
- R2 공개 URL은 현재 `R2_PUBLIC_BASE_URL` 기준으로 확인했다.
