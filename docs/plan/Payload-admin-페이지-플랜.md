## 1. 결론 요약

관리자 페이지는 Payload의 컬렉션 목록을 그대로 노출하는 방식이 아니라, **센터별 운영 메뉴**를 기준으로 정리한다.

핵심 기준은 아래와 같다.

- 관리자는 센터별로 1명씩 있다고 가정한다.
- 센터는 `art`, `exam`, `highteen`, `kids`, `avenue` 5개다.
- `master`, `admin`, `manager` 3단계 역할을 둔다.
- 아트센터 관리자는 `admin` 역할을 가지며, 센터 관리자인 `manager`보다 높은 중간 계급으로 둔다.
- 아트센터 관리자는 모든 센터 콘텐츠와 전사 뉴스 노출을 관리할 수 있다.
- 나머지 센터 관리자는 자기 센터에 연결된 메뉴와 콘텐츠만 관리한다.
- 공개 사이트의 메뉴 구조를 먼저 기준으로 삼고, Payload 컬렉션은 그 메뉴를 운영하기 위한 데이터 저장소로 본다.

즉, 관리자 메뉴명은 개발자용 컬렉션명이 아니라 운영자 언어로 보여야 한다.

## 2. 관리자 역할

| 역할 | 센터 | 권한 등급 | 권한 범위 |
|---|---|---|---|
| `master` | 전체 | 100 | 시스템/권한/전체 콘텐츠 관리 |
| `admin` | 아트센터 | 80 | 전체 센터 콘텐츠 조회/수정, 전사 뉴스 노출 관리, 공통 배너/홈 노출 관리 |
| `manager` | 입시센터 | 50 | 입시센터 메뉴와 콘텐츠만 조회/수정 |
| `manager` | 하이틴센터 | 50 | 하이틴센터 메뉴와 콘텐츠만 조회/수정 |
| `manager` | 키즈센터 | 50 | 키즈센터 메뉴와 콘텐츠만 조회/수정 |
| `manager` | 애비뉴센터 | 50 | 애비뉴센터 원페이지 섹션과 관련 콘텐츠만 조회/수정 |

### 2-1. 권한 구현 방향

Payload `users` 컬렉션에 최소한 아래 필드가 필요하다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `name` | text | 관리자 이름 |
| `email` | email | 로그인 계정 |
| `role` | select | `master`, `admin`, `manager` |
| `permissionLevel` | number | `100`, `80`, `50`처럼 권한 등급 비교에 사용 |
| `center` | select | `art`, `exam`, `highteen`, `kids`, `avenue` |

권한 규칙은 아래처럼 잡는다.

- `role=master`는 모든 컬렉션, 모든 센터 데이터, 사용자/권한 설정에 접근할 수 있다.
- `role=admin`은 아트센터 관리자이며, 콘텐츠 관리에서는 전체 센터 접근 권한을 가진다.
- `role=manager`는 자기 `center`와 매칭되는 데이터만 접근할 수 있다.
- 센터 공통 콘텐츠와 전사 노출 콘텐츠는 `master` 또는 `admin`만 수정한다.
- 센터 관리자는 공통 콘텐츠를 읽을 수는 있지만 수정하지 않는다.

권한 등급을 두는 이유는 역할명이 늘어나도 access 로직을 단순하게 유지하기 위해서다.
예를 들어 전체 콘텐츠 수정은 `permissionLevel >= 80`, 사용자 권한 관리는 `permissionLevel >= 100`처럼 나눌 수 있다.
`manager`는 센터 운영자 권한명으로만 쓰고, 캐스팅 매니저 콘텐츠에는 쓰지 않는다.

## 3. Admin 메뉴 설계 원칙

### 3-1. 운영자 기준 메뉴

Payload Admin 사이드바는 아래처럼 운영자 작업 단위로 묶는다.

| 그룹 | 목적 |
|---|---|
| 센터 홈/노출 | 센터 홈, 배너, 주요 노출 콘텐츠 |
| 교육/강사진 | 강사, 교육진, 강사 대표작, 커리큘럼 |
| 공지/뉴스 | 센터별 공지, 전사 노출 뉴스 |
| 캐스팅/오디션 | 캐스팅 디렉터, 캐스팅 포트폴리오, 출연현황, 오디션 CTA |
| 프로필/성과 | 배우 프로필, 루키, 출신 아티스트, 아역배우 프로필 |
| 작품/활동 | 라인업, 영화, 드라마, 촬영, 연출, 스타카드 |
| 후기/합격 | 후기, 합격 후기, 합격 영상, 합격 현황 |
| 상담/폼 | 상담신청, 온라인 상담현황, 오디션 지원 |
| 시스템 | 사용자, 레거시 추적, 마이그레이션 확인 |

### 3-2. 컬렉션명 정책

현재 일부 컬렉션은 영어 라벨이거나 라벨이 없다.

아래 컬렉션은 우선 라벨 정리가 필요하다.

| 컬렉션 | 현재 상태 | 권장 라벨 |
|---|---|---|
| `teachers` | `Teachers` | 강사 |
| `agencies` | 라벨 없음 | 위탁교육 기업 |
| `profiles` | 라벨 없음 | 프로필 |
| `news` | 라벨 없음 | 공지/뉴스 |
| `castings` | 라벨 없음 | 캐스팅 디렉터 |
| `banners` | 라벨 있음, 그룹이 `레거시 콘텐츠` | 배너 |
| `video-castings` | 라벨 있음, 그룹이 `레거시 콘텐츠` | 캐스팅 포트폴리오 |

`sourceTable`, `sourceId`, `legacyMeta`는 운영자용 주요 필드가 아니다.
상세 화면 하단의 접힌 "레거시 정보" 영역으로 이동하는 것이 맞다.

### 4-1. Rename 후보

혼선을 줄이기 위해 아래 rename을 권장한다.

| 현재명 | 권장명 | 이유 |
|---|---|---|
| `castings` | `casting-directors` | 실제 데이터가 캐스팅 게시물이 아니라 캐스팅 디렉터/매니저 이력이다. `manager`는 권한 role과 충돌하므로 컬렉션명에는 쓰지 않는다. |
| `teacher-files` | `teacher-filmographies` | 실제 데이터가 첨부파일이 아니라 강사의 대표작 포스터와 작품명이다. 운영자 메뉴명은 "강사 대표작"으로 둔다. |
| `teacher_lessons` | `curriculums` | 실제 의미가 강사별 커리큘럼/수업 내용이다. |
| `artist_press` | `artist-press` 또는 `alumni-artists` | 출신 아티스트 메뉴의 정보다. 공개 메뉴명 기준으로는 "출신 아티스트"가 더 명확하다. |
| `casting_appearances` | `casting-appearances` | 진행중인 캐스팅 출연현황. `screen_appearances`와 분리 유지. |
| `screen_appearances` | `screen-appearances` | 드라마/광고 출연장면. `casting_appearances`와 분리 유지. |

MariaDB work table명은 당장 바꾸지 않는다.
`teacher_files`는 레거시 통합/검증 파이프라인의 중간 산출물이므로 추적성을 위해 유지하고, Payload 이관 시점에 운영 도메인명인 `teacher-filmographies`로 받는다.

## 5. 개발 대상 메뉴만 추린 공개 메뉴 - DB 연결 초안

정적 안내 페이지는 Payload에서 관리하지 않고 코드 기반으로 관리한다.
아래 표에서 "코드 기반 정적 페이지"로 표시한 항목은 개발 대상이지만 DB 컬렉션과 직접 연결하지 않는다.

## 8. 데이터 필터링 정책 초안

센터 관리자 접근 제어를 하려면 대부분의 공개 콘텐츠 컬렉션에 센터 필드가 필요하다.

### 8-1. 권장 센터 필드

| 필드 | 타입 | 사용처 |
|---|---|---|
| `center` | select | 단일 센터 콘텐츠 |
| `centers` | select hasMany | 여러 센터에 동시에 노출되는 콘텐츠 |
| `isGlobal` | checkbox | 전사 공통 노출 |
| `displayStatus` | select | `draft`, `published`, `archived` |
| `featured` | checkbox | 센터 홈/게이트 노출 |

### 8-2. 접근 규칙

- 마스터 관리자는 `center`, `centers`, `isGlobal`과 무관하게 전체 접근 가능.
- 센터 관리자는 `center=자기센터` 또는 `centers`에 자기 센터가 포함된 문서만 접근 가능.
- `isGlobal=true` 문서는 센터 관리자가 읽을 수는 있지만 수정은 마스터만 가능하게 두는 것이 안전하다.
- 센터 관리자가 전사 뉴스로 올리는 기능은 두지 않는다. 필요하면 마스터가 승인/변경한다.
- 휴무일 안내처럼 모든 센터에 같은 운영방침을 공지해야 하는 경우 `isGlobal=true`를 사용한다.
- 글로벌 뉴스는 대부분 운영방침성 게시글로 제한한다.

## 9. 확정 정책과 남은 질문

아래는 이번 논의에서 확정한 정책이다.

1. `master`, `admin`, `manager` 3단계 역할을 둔다.
   - `admin`은 아트센터 관리자이며, 콘텐츠 관리에서는 전체 센터 권한을 가진다.
   - 사용자/권한 자체를 관리하는 최상위 권한은 `master`에 둔다.

2. 센터 공통 콘텐츠는 센터 관리자가 수정하지 않는다.
   - 예: 회사소개, 학원 100% 이용법, 스타카드 멤버십서비스.
   - 코드 기반 정적 페이지 또는 마스터/아트센터 관리자 관리 영역으로 둔다.
   - 이 방식이 편한 이유는 책임 소재가 명확하고, 같은 문구를 여러 센터 관리자가 서로 다르게 고치는 일을 막을 수 있기 때문이다.
   - 단점은 센터별 작은 문구 수정도 마스터/개발 작업을 거쳐야 한다는 점이다.

3. `NEWS & NOTICE`의 전사 노출 뉴스는 `news`에 `isGlobal`, `featured`, `centers`를 둔다.
   - 마스터는 전사 노출 설정 가능, 센터 관리자는 자기 센터 뉴스만 작성.
   - 휴무일 안내 같은 운영방침성 게시글이 주된 글로벌 뉴스다.

4. 애비뉴센터는 단순 랜딩으로 시작한다.
   - 형식상 메뉴와 관리자 계정은 둘 수 있다.
   - 실제 콘텐츠 운영은 초기에는 아트센터가 대행할 수 있다.

5. 상담/오디션 신청 데이터는 신규 접수만 받는다.
   - 레거시 상담 데이터는 이관하지 않는다.

6. 정적 페이지는 코드 기반으로 관리한다.
   - `site-pages` 컬렉션은 만들지 않는다.
   - FAQ는 내용이 적어도 신규 DB를 만든다.
   - FAQ 레거시 데이터는 가져오지 않는다.

남은 질문:

1. `video-castings`는 활성 메뉴에 포함하되, 공개 메뉴 구조도에서 이름을 확정해야 한다. 현재 문서에서는 `캐스팅 포트폴리오`로 둔다.

확정된 마이그레이션 정책:

- 로컬 Postgres는 초기화 가능하다. 현재 운영 데이터가 없으므로 clean slate 기준으로 진행한다.
- `castings`의 최종 영문 컬렉션명은 `casting-directors`로 한다. `manager`는 권한 role과 충돌하므로 피한다.
- `teacher-files`의 최종 영문 컬렉션명은 `teacher-filmographies`로 한다.
- `artist_press`의 최종 영문 컬렉션명은 `artist-press`로 한다.
- 기존 c0 컬렉션(`appearances`, `shoots`, `reviews` 등)은 MariaDB work table 기준 컬렉션으로 대체/통합한다.
- 센터 필드는 모든 공개 콘텐츠 컬렉션에서 복수 센터 처리가 가능한 `centers` 기준으로 둔다.
- 이미지 경로는 우선 로컬 다운로드/레거시 경로를 보존하고, R2/Blob URL 단순화는 후속 단계로 분리한다.
