# 마이그레이션 정제 체크리스트

> 목적: 로컬 Postgres/Payload DB에서 정제가 끝난 테이블을 확인하고, 원격 Vercel Neon DB 이전 전에 누락을 막는다.
> 마지막 갱신: 2026-05-12

## 현재 순서 판단

사용자가 정리한 순서는 큰 방향이 맞다.

```text
1. 원격/원본 dump 확보
2. 로컬 MariaDB에 dump 복원
3. 로컬 MariaDB work table을 로컬 Postgres/Payload DB로 적재
4. 로컬 Postgres/Payload DB 정제 및 검증
5. 원격 Vercel Neon DB로 schema/data 이전
6. 완료
```

주의할 점은 3번이다. 현재 저장소 기준으로는 MariaDB 원본 DB를 PostgreSQL로 그대로 변환하는 단계가 아니라, `bnb_legacy_work.*` work table을 Payload 컬렉션 구조에 맞게 정제 적재하는 단계로 본다.

5번은 4번의 로컬 검증이 끝난 뒤에만 진행한다. 원격 Neon에는 검증된 migration과 데이터만 반영한다.

## 4번 정제 완료 기준

각 테이블은 아래 항목을 모두 확인한 뒤에만 `정제 완료`로 체크한다.

- 로컬 Postgres/Payload 적재가 완료됐다.
- 정제 전 원본 추적 값(`sourceDb`, `sourceTable`, `sourceId` 또는 동등한 값)을 확인했다.
- 정제 완료 후 원본 추적 값을 제거했다면 migration과 backfill/검증 리포트로 재현 경로가 남아 있다.
- 필수 운영 필드가 비어 있지 않다.
- 센터 매핑이 맞다.
- 공개/비공개 상태 값이 맞다.
- 레거시 게시 날짜가 있는 컬렉션은 `published_at`과 Payload `created_at`이 정렬 기준에 맞게 일치한다.
- 대표 이미지, 로고, 본문 이미지, 첨부 이미지 경로 또는 media relation이 확인됐다.
- `/test/postgres/*` 검수 화면에서 샘플이 정상 표시된다.
- MariaDB work table 카운트와 Postgres/Payload 카운트 차이를 설명할 수 있다.
- 정제 과정에서 의도적으로 제외한 레코드는 사유가 기록됐다.

## 공통 자산 체크

- [ ] `media` relation: Payload 문서가 로컬 파일 경로 문자열만 보지 않고 필요한 곳에서 `media` relation을 참조한다.
- [ ] `public/legacy/*` staging: 로컬 검수에 필요한 legacy 이미지가 존재하고, 누락 파일은 리포트 또는 TODO로 남겼다.
- [ ] R2 object key 정책: DB에는 개발자 계정 R2 완성 URL을 저장하지 않고, object key 또는 Payload media relation 기준으로 정리했다.
- [ ] 레거시 fallback: 신규 필드가 비어 있어도 기존 레거시 값으로 목록/상세/SEO가 깨지지 않는다.
