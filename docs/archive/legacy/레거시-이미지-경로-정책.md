# 레거시 이미지 경로 정책

## 결정

레거시 이미지 경로는 두 단계로 나누어 관리한다.

1. MariaDB work table과 `public/legacy`는 원본 추적과 재다운로드를 위한 staging/cache로 둔다.
2. PostgreSQL/Payload 적재 단계에서 서비스용 Cloudflare R2 object key 또는 Media relation으로 정리한다.

운영 원격 저장소는 `Cloudflare R2`로 확정한다. `public/legacy`는 최종 저장소가 아니라 R2 업로드 전 로컬 staging/cache다.

개발 순서는 `개발자 계정 R2 업로드/검증 -> 클라이언트 계정 R2로 객체 복사 -> 환경변수 교체`를 기본으로 한다. 이 이전 절차가 쉬워야 하므로 DB와 콘텐츠 본문에는 개발자 계정에 종속된 전체 URL을 저장하지 않는다.

## 배경

레거시 원본은 센터별 MariaDB에 나뉘어 있다.

- `baewoo`
- `bnbuniv`
- `kidscenter`
- `bnbhighteen`

각 센터 DB에는 같은 성격의 테이블이 반복되며, `wr_id`, `bn_id`, `bo_table`, 파일명이 서로 충돌할 수 있다. 따라서 파일 경로를 너무 일찍 예쁘게 바꾸면 원본 추적, 중복 판정, FTP 재다운로드가 어려워진다.

## MariaDB Work Table 원칙

MariaDB work table은 최종 서비스 DB가 아니라 원본을 검증 가능한 형태로 모은 staging이다.

따라서 work table에는 아래 정보를 보존한다.

- `source_db`
- `source_table`
- `source_id`
- `bo_table`
- 원본 파일명
- 원본 FTP/legacy path
- 다운로드된 `public/legacy` path
- `legacy_meta`

`public/legacy` 경로는 출처가 드러나도록 길어도 괜찮다.

```text
public/legacy/news/baewoo/new_notice/123/file-0/foo.jpg
public/legacy/casting-appearances/kidscenter/new_appear/149/thumbnail/bar.png
public/legacy/exam-results/bnbuniv/victory10/77/thumbnail/baz.jpg
```

이 단계의 좋은 경로는 사람이 보기 좋은 경로가 아니라, 원본을 다시 찾을 수 있는 경로다.

## PostgreSQL/Payload 적재 원칙

PostgreSQL/Payload는 서비스용 데이터이므로 이 단계에서 경로를 정리한다.

권장 방식:

- Payload Media 컬렉션 또는 media relation으로 연결한다.
- R2 object key는 slug/source 기반으로 정규화한다.
- 공개 URL은 저장하지 않고 `R2_PUBLIC_BASE_URL + objectKey`로 조합한다.
- 대표 이미지, 본문 이미지, 로고, 학생 이미지처럼 역할을 분리한다.
- 중복 파일은 checksum 또는 원본 key 기준으로 dedupe한다.
- 원본 출처는 삭제하지 않고 `legacy_meta`에 보존한다.

예시:

```text
public/legacy/news/baewoo/new_notice/123/file-0/foo.jpg
```

Payload 적재 후:

```text
R2 objectKey: media/news/news-baewoo-123/file-0.jpg
publicUrl: ${R2_PUBLIC_BASE_URL}/media/news/news-baewoo-123/file-0.jpg
```

또는:

```text
news.heroImage -> media document
media.objectKey -> media/news/news-baewoo-123/file-0.jpg
news.legacyMeta.originalPath -> public/legacy/news/baewoo/new_notice/123/file-0/foo.jpg
```

## 하지 않을 것

- MariaDB 원본 DB의 파일 경로를 수정하지 않는다.
- MariaDB work table에서 원본 추적 정보를 덮어쓰지 않는다.
- `public/legacy`를 최종 서비스 경로처럼 다듬지 않는다.
- 파일명을 사람이 보기 좋게 바꾸면서 원본 파일명과 출처를 잃지 않는다.
- DB/Payload에 개발자 계정의 `r2.dev` URL이나 임시 커스텀 도메인 전체 URL을 저장하지 않는다.
- 본문 HTML에 `${R2_PUBLIC_BASE_URL}`이 적용된 완성 URL을 영구 치환하지 않는다. 본문 치환이 필요하면 objectKey 기반 매핑을 별도로 유지한다.

## 실행 흐름

```text
센터별 MariaDB 원본
  -> bnb_legacy_work.* work table
  -> FTP dry-run
  -> public/legacy/* 다운로드
  -> PostgreSQL/Payload 적재
  -> R2 object key 또는 media relation 생성
  -> 개발자 계정 R2 업로드/검증
  -> 클라이언트 계정 R2로 동일 objectKey 복사
  -> R2_PUBLIC_BASE_URL 및 계정 환경변수 교체
```

## Payload Admin UI 방향

Payload admin은 SQL 테이블 뷰어처럼만 쓰는 도구가 아니다. 컬렉션 설정과 커스텀 컴포넌트로 CMS다운 편집 UI를 만들 수 있다.

가능한 구성:

- 필드 그룹화: 탭, 그룹, collapsible 영역
- 관계형 필드: Media, Teachers, Centers 같은 relation
- 업로드/이미지 필드: 썸네일, alt, caption 관리
- 조건부 필드: 센터나 콘텐츠 타입에 따라 필요한 필드만 노출
- 커스텀 Cell/Field 컴포넌트: 목록에서 이미지 미리보기, 상태 배지, 원본 링크 표시
- Admin description/help text: 편집자용 안내 문구
- access control: 역할별 읽기/쓰기 제한
- hooks: 저장 전 slug 생성, 이미지 정규화, legacy_meta 보존

현재 admin이 SQL 테이블 뷰어처럼 보이는 이유는 마이그레이션 검증 단계에서 원본 필드를 그대로 노출하는 구조에 가깝기 때문이다. 최종 CMS 단계에서는 컬렉션별로 편집자 업무 흐름에 맞춘 Admin UI를 별도로 설계해야 한다.

## 권장 후속 작업

1. Media 컬렉션 정책 확정
2. `public/legacy` -> R2/Payload Media 적재 스크립트 설계
3. 컬렉션별 이미지 역할 정의
4. Payload admin 필드 그룹/탭 구조 설계
5. legacy source 추적 필드를 read-only 영역으로 분리

R2 임시 계정 사용과 클라이언트 계정 이전 기준은 `docs/R2-스토리지-임시계정-이전-정책.md`를 따른다. 운영 저장소 결정은 R2이며, Vercel Blob은 기본안에서 제외한다.
