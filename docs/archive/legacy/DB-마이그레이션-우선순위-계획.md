# 배우앤배움 DB 마이그레이션 우선순위 계획

> 작성일: 2026-04-08
> 목적: `data/baewoo.sql` 기준으로, 전체 이관이 아닌 `필수 데이터 선별 + 순차 테스트 기반 마이그레이션` 기준을 정리한다.

---

## 1. 결론 요약

`data/baewoo.sql` 전체를 그대로 `Vercel Marketplace Postgres` 계열 DB로 이관하는 방식은 권장하지 않는다.

이유는 아래와 같다.

- 현재 덤프는 `MariaDB 10.1 + phpMyAdmin + MyISAM` 기반의 구형 MySQL 덤프다.
- 전체 테이블 수가 많고(`135개`), 게시판성 `g5_write_*` 테이블이 과도하게 세분화되어 있다(`69개`).
- 로그/임시저장/회원/우편번호/백업 테이블까지 포함되어 있어 새 프로젝트에 불필요한 데이터가 많다.
- `0000-00-00`, `set`, `enum`, `AUTO_INCREMENT`, `UNSIGNED` 등 Postgres 직이관에 맞지 않는 문법과 데이터가 존재한다.
- 상담/회원 데이터에는 개인정보가 포함되어 있어 기술 검토와 별도로 운영 정책 검토가 필요하다.

따라서 권장 방향은 아래와 같다.

1. `공개 사이트 운영에 꼭 필요한 데이터`부터 선별한다.
2. `읽기 전용 콘텐츠`를 먼저 옮긴다.
3. `상담/운영 데이터`는 나중에 별도 검토 후 이관한다.
4. 각 단계마다 소량 테스트 후 다음 단계로 넘어간다.

---

## 2. 현재 덤프의 특징

### 기술적 특징

- 소스 DB: `MariaDB 10.1`
- 덤프 형태: `phpMyAdmin SQL Dump`
- 스토리지 엔진: 전 테이블 `ENGINE=MyISAM`
- 외래키: 거의 없음
- 날짜 기본값: `0000-00-00`, `0000-00-00 00:00:00` 다수
- MySQL 전용 타입: `set(...)`, `enum(...)`, `UNSIGNED`, `AUTO_INCREMENT`

### 운영 관점 특징

- 그누보드 기반 `content`, `board`, `write_*` 구조라서 새 프로젝트의 정보구조와 1:1 대응시키기 어렵다.
- 하나의 개념이 여러 테이블로 흩어져 있다.
  - 예: 캐스팅, 프로필, 공지, 상담이 각각 별도 `g5_write_*` 테이블
- 동일 성격의 백업 테이블이 다수 존재한다.
  - 예: `*_bak`
- 시스템 운영 흔적 테이블이 포함되어 있다.
  - 예: `g5_visit`, `g5_uniqid`, `g5_autosave`
- 회원 데이터에는 스팸/노이즈로 보이는 계정이 섞여 있다.

## 3. 프로젝트 기준으로 실제 필요한 데이터

기존 기획 문서를 기준으로 현재 프로젝트에서 중요한 콘텐츠는 아래 순서다.

### 가장 먼저 필요한 영역

- 공지사항/뉴스
- 강사진
- 센터 소개 및 고정 페이지
- 캐스팅/오디션
- 프로필/성과

### 그 다음 필요한 영역

- 통합 상담 신청/상담 현황
- 운영용 FAQ/이용안내
- 센터별 프로그램/부가 운영 콘텐츠

### 굳이 가져오지 않아도 되는 영역

- 구 사이트 운영 로그
- 임시저장 데이터
- 우편번호 마스터
- 사용하지 않을 백업 테이블
- 기존 회원 계정 전체

이 우선순위는 아래 문서와 맞물린다.

- 공개 사이트 핵심 IA: `공지`, `강사진`, `캐스팅`, `프로필`, `상담`
- 신규 URL 구조: `/news`, `/[center]/faculty`, `/[center]/casting`, `/[center]/contact`
- 클라이언트 요구: `강사 소개`, `공지사항`, `갤러리`, `상담` 중심의 직접 관리형 CMS

|---|---|---|---|---|
| `g5_content` | 공통/정적 페이지 본문(`co_id`) | 소개, 시설안내, 오시는 길, FAQ, 이용안내 등 | P0 | 고정 페이지 마이그레이션의 핵심 |
| `g5_content2` | 보조 정적 페이지 본문 | 센터별/추가 정적 페이지 | P0 | `g5_content`와 함께 검토 |
| `g5_teacher` | 강사진/교육진 상세 | `/[center]/faculty`, `/faculty` | P0 | 대표 강사진 데이터 |
| `g5_teacher2` | 추가 강사진/센터별 강사진 | 센터별 강사진 보강 | P0 | `g5_teacher`와 통합 설계 필요 |
| `g5_write_new_notice` | 공지/뉴스 게시물 | `/news`, `/[center]/results` 일부 | P0 | 공지/뉴스 운영의 핵심 |

| `g5_write_new_casting2` | 드라마 캐스팅(유캐스팅) | `/[center]/casting` | P1 | 표준 게시판형 |

| 소스 테이블 | 담고 있는 데이터 | 신규 프로젝트 활용 예상 | 우선순위 | 메모 |

| `g5_visit`, `g5_visit_sum`, `g5_popular`, `g5_uniqid` | 방문/통계/식별값 | 제외 | 신규 서비스에서 재사용 불필요 |
| `zipcode` | 우편번호 마스터 | 제외 | 구주소 체계 기반, 별도 서비스로 대체 가능 |

## 5. 단계별 이관 전략

이번 마이그레이션은 `한 번에 끝내는 프로젝트`가 아니라 `단계별 검증 프로젝트`로 보는 것이 안전하다.

### 0단계. 기준 모델 확정

- 새 서비스의 콘텐츠 모델을 먼저 정한다.
- `source table -> target collection/table` 매핑을 확정한다.
- 여기서 중요한 기준은 `원본 구조를 복제`하는 것이 아니라 `새 프로젝트의 IA에 맞게 재구성`하는 것이다.

권장 타깃 개념은 아래 수준으로 단순화한다.

- `pages`
- `faculty`
- `news`
- `casting`
- `profiles`
- `centers`
- `inquiries`

### 1단계. 읽기 전용 콘텐츠부터 테스트 이관

대상:

- `g5_content`
- `g5_content2`
- `g5_teacher`
- `g5_teacher2`
- `g5_write_new_notice`

이 단계 목표:

- Postgres 스키마/컬렉션 구조 검증
- HTML 본문 정리 규칙 검증
- slug 생성 규칙 검증
- 센터 매핑 방식 검증

이 단계는 개인정보가 거의 없고, 공개 페이지 검증에 바로 쓸 수 있어 가장 먼저 진행하는 것이 좋다.

### 2단계. 캐스팅/프로필 콘텐츠 이관

대상:

- `g5_write_new_profile`
- `g5_write_new_casting*`
- `g5_agency`
- `g5_plan`

이 단계 목표:

- 센터별 콘텐츠 분류 규칙 확정
- 갤러리형/게시판형 콘텐츠를 하나의 새 모델로 통합할 수 있는지 검증
- 이미지 경로/외부 링크/에디터 HTML 정리 검증

### 3단계. 상담 데이터 분리 검토 후 이관

대상:

- `g5_write_new_counsel`
- `g5_write_new_counsel_2`
- `sm_customer`

이 단계 목표:

- 개인정보 보존 필요 여부 확인
- 실제로 가져갈 컬럼만 선별
- 관리자 검색/상태관리 기준 재정의

이 단계는 무조건 후순위가 맞다.

이유:

- 새 프로젝트는 `통합 상담 신청 시스템`으로 개편 예정이라 기존 상담 데이터 구조를 그대로 복제할 이유가 적다.
- 이름, 휴대폰, 생년월일, 성별 등 민감 정보가 포함되어 있다.
- 오래된 상담 데이터는 운영 가치보다 리스크가 클 수 있다.

## 6. 분리 익스포트 권장 순서

전체 SQL 파일 하나보다, 아래 순서대로 묶어서 분리 익스포트받는 편이 훨씬 낫다.

### 1차 익스포트

- `g5_content`
- `g5_content2`
- `g5_teacher`
- `g5_teacher2`
- `g5_write_new_notice`

용도:

- 가장 먼저 새 DB에 올려볼 테스트 세트
- 공개 사이트 뼈대를 만드는 데 필요한 최소 데이터

### 2차 익스포트

- `g5_write_new_profile`
- `g5_write_new_casting`
- `g5_write_new_casting2`
- `g5_write_new_casting3`
- `g5_write_new_casting_abio`
- `g5_write_new_casting_bx`
- `g5_agency`
- `g5_plan`

용도:

- 성과/활동/캐스팅 섹션 고도화

### 3차 익스포트

- `g5_write_new_counsel`
- `g5_write_new_counsel_2`
- `sm_customer`

용도:

- 상담 이력 마이그레이션 여부 검토

### 참고용 익스포트

- `g5_menu`

용도:

- 메뉴/IA 참고
- 실제 데이터 이관 대상은 아님

## 7. 우선순위별 실행 권장안

### 권장안 A. 가장 현실적인 시작점

1. `정적 페이지 + 강사진 + 공지`만 먼저 옮긴다.
2. 새 공개 사이트에서 페이지 렌더링과 검색 구조를 먼저 검증한다.
3. 그다음 `캐스팅 + 프로필`로 확장한다.
4. 마지막에 `상담`을 붙인다.

이 방식이 가장 안전하다.

### 권장안 B. 당장 제외해도 되는 데이터

아래는 초기에 아예 빼고 시작해도 된다.

- 회원 전체
- 방문 로그/통계
- 임시저장
- 우편번호
- 백업 테이블
- 그누보드 엔진 메타

## 8. 다음 액션 제안

문서 기준으로 실제 작업은 아래 순서가 좋다.

1. `1차 익스포트 대상 테이블`을 확정한다.
2. 각 테이블의 `source column -> target field` 매핑표를 만든다.
3. `P0 테이블만` 대상으로 Postgres용 테스트 스키마를 만든다.
4. 소량 샘플 import 후 화면 렌더링을 검증한다.
5. 문제가 없으면 2차 이관으로 넘어간다.

가장 먼저 착수할 작업은 `P0 테이블 매핑표 작성`이다.

## 9. 요약

이번 덤프는 `전체 이전`보다 `선별 이전`이 맞다.

- 먼저 가져올 것: `정적 페이지`, `강사진`, `공지`
- 다음에 가져올 것: `캐스팅`, `프로필`, `에이전시/성과`
- 나중에 검토할 것: `상담`, `고객`, `운영이력`
- 가져오지 않을 것: `회원`, `로그`, `임시저장`, `우편번호`, `백업`, `그누보드 시스템 메타`

핵심 원칙은 하나다.

`원본 DB를 그대로 재현하지 말고, 새 프로젝트에 필요한 콘텐츠만 작게 옮기면서 단계별로 검증한다.`
