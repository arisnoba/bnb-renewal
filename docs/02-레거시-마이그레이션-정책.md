# 레거시 마이그레이션 정책

> 목적: 레거시 DB, FTP 이미지, Payload/Postgres/R2 이관 기준을 한 문서로 관리한다.
> 마지막 갱신: 2026-04-28

## 기본 방향

현재 이관 기준은 과거의 P0/P1/P2 또는 C0 단계가 아니다. 해당 기준은 폐기된 계획으로 보고 현재 작업 판단에 사용하지 않는다.

현재 기준 파이프라인은 아래 하나다.

```text
data/legacy_dumps
  -> 로컬 MariaDB 복원
  -> 로컬 MariaDB work table 구성
  -> 로컬 Postgres/Payload DB에 정제 적재
  -> 로컬 검증
  -> Vercel Neon으로 migration/데이터 이전
```

대량 write는 dry-run과 샘플 검증 후 명시적으로만 진행한다.

## 데이터 흐름

```text
data/legacy_dumps의 센터별 MariaDB dump
  -> 로컬 MariaDB 복원
  -> bnb_legacy_work.* work table
  -> FTP/legacy 이미지 dry-run
  -> public/legacy/* staging/cache
  -> 로컬 Postgres/Payload DB 정제 적재
  -> Payload media relation 또는 R2 objectKey 연결
  -> 로컬 검증
  -> Vercel Neon으로 migration/데이터 이전
  -> 개발자 계정 R2 업로드/검증
  -> 클라이언트 계정 R2로 동일 objectKey 복사
```

## 원본 보존 원칙

MariaDB work table과 `public/legacy`는 최종 서비스 경로가 아니라 원본 추적용 staging/cache다.

보존해야 하는 값:

- `source_db`
- `source_table`
- `source_id`
- `bo_table`
- 원본 파일명
- 원본 FTP/legacy path
- 다운로드된 `public/legacy` path
- `legacy_meta`

원본 DB, work table, legacy path는 임의로 덮어쓰지 않는다.

## 이미지 이관 원칙

- `public/legacy`는 R2 업로드 전 로컬 cache로만 본다.
- 서비스용 이미지는 Payload `media` relation 또는 R2 `objectKey`로 연결한다.
- DB/Payload에는 개발자 R2 완성 URL을 저장하지 않는다.
- 대표 이미지, 본문 이미지, 로고, 학생 이미지처럼 역할을 분리한다.
- 대량 write와 대량 media 생성은 dry-run 결과 확인 후 명시적으로만 수행한다.

현재 적용된 media 연결:

- `news.thumbnailPath` -> `thumbnailMedia`, `meta.image` dry-run 스크립트
- `artist-press.thumbnailPath` -> `thumbnailMedia`, `meta.image` dry-run 스크립트
- `artist-press.agencyLogoPath` -> `agencyLogoMedia` dry-run 스크립트
- `exam-school-logos.logoPath` -> `logoMedia` 실제 연결 완료

## 주요 명령

```bash
npm run legacy:db:import
npm run legacy:db:verify
npm run legacy:work:news
npm run legacy:work:artist-press
npm run payload:legacy:link-news-thumbnails
npm run payload:legacy:link-artist-press-media
npm run payload:legacy:link-exam-school-logo-media
```

실제 DB 쓰기는 각 스크립트의 `--write` 옵션이 있는 경우에만 수행한다.

## 로컬 Postgres에서 Neon으로 이전

정제와 검증은 로컬 Postgres/Payload DB에서 먼저 끝낸다. Vercel Neon은 로컬에서 검증된 schema와 데이터를 옮기는 대상으로 본다.

기준:

- 로컬 MariaDB는 원본 복원과 work table 생성용이다.
- 로컬 Postgres는 Payload 컬렉션 기준으로 정제된 데이터의 1차 정본이다.
- Neon 이전 전에는 schema diff, 샘플 레코드, media relation, legacy fallback을 확인한다.
- Neon에는 검증된 migration과 데이터만 반영한다.
- 운영 R2 이전과 Neon 이전은 분리해서 검증한다.

## Migration 주의

Payload migration 생성 전에는 현재 DB schema diff를 먼저 확인한다. Payload가 rename/create를 물을 때 자동 선택하면 기존 컬럼을 잘못 rename할 수 있다.

특히 레거시 필드는 삭제하지 않는다. 신규 UX 필드는 optional로 추가하고, 프론트는 신규 필드 우선 fallback으로 렌더링한다.
