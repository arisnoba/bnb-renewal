# 레거시 테이블 분류 초안

> 기준: 로컬 MariaDB `baewoo`, `bnbuniv`, `kidscenter`, `bnbhighteen` 복원본  
> 작성일: 2026-04-21  
> 목적: GUI 확인 전에 버릴 테이블과 통합할 테이블의 1차 후보를 좁힌다.

## 결론

GUI는 필요하다. 다만 모든 테이블을 눈으로 훑기보다, 아래 분류를 기준으로 **통합 후보부터 확인**하고 나머지는 필요할 때만 보는 방식이 효율적이다.

우선 이관은 공개 사이트에 노출될 콘텐츠 중심으로 진행한다.

1. 티쳐/강사진
2. 공지/뉴스
3. 프로필/합격자/포트폴리오
4. 캐스팅/에이전시/제휴
5. 배너/메인 이미지/팝업
6. 라인업/영상/작품/출연/연출/촬영/후기/스타카드

상담, 회원, 포인트, 방문 로그, 우편번호, SMS, 자동저장, 인증/로그인 계열은 초기 공개 사이트 이관 범위에서 제외한다.

| --- | --- | --- |
| 티쳐 | `g5_teacher`, `g5_teacher2`, `g5_teacher_file`, `g5_lesson_teacher` | 최우선. 센터별 출처를 붙여 통합한다. |
| 공지/뉴스 | `g5_write_new_notice`, `g5_write_notice`, `g5_write_bnb_highteen_news`, `g5_write_bnb_highteen_news2`, `g5_write_news_acdm` | 센터별 공지/뉴스로 통합한다. |
| 프로필/합격자 | `g5_write_new_profile`, `g5_write_profile`, `g5_write_portfolio_profile`, `g5_write_new_profile2`, `g5_write_new_profile3`, `g5_write_qr_profile` | 성격이 섞여 있어 GUI로 샘플 확인 후 컬렉션을 나눈다. |

| 라인업 | `g5_write_lineup`, `g5_write_new_lineup`, `g5_write_qr_lineup` | 통합 사이트의 lineups 후보. |

## 샘플 확인 메모

- `g5_teacher2`는 `summary`, `message`, `piece`, `bn_bimg`, `photo_img1~6`, `it_img1~8` 구조다. `message`에는 HTML 경력 테이블이 들어 있다.
- `g5_board_file`은 `bo_table`, `wr_id`, `bf_file`, `bf_source`로 원본 게시글 첨부를 연결한다. 예: `bo_table = new_notice`, `wr_id = 3177`.
- `g5_write_new_casting` 샘플은 배우 이름이 제목으로 들어간 캐스팅 성과성 콘텐츠다.
- `bnbuniv.g5_write_victory*`는 “동국대학교 최종합격 임태성”처럼 입시 합격자 콘텐츠다. 프로필과 분리하거나 success/victory 계열 컬렉션으로 둘 수 있다.
- `kidscenter.g5_write_new_profile`은 2026년 최신 키즈 프로필성 콘텐츠가 들어 있어, 기존 `profiles` 이관에서 누락되면 안 된다.

## 보류 후보

아래 테이블은 데이터가 있지만 공개 사이트 핵심 이관에는 바로 넣지 않는다.

| 테이블 | 이유 |
| --- | --- |
| `g5_board`, `g5_board_new` | 게시판 메타/최근글 성격. 원본 게시판 매핑 확인용으로만 사용. |
| `g5_board_file` | 첨부파일 매핑에 중요하지만 독립 컬렉션이 아니라 각 게시글 테이블과 조인해서 사용. |
| `g5_write_new_audition` | 공개 모집/오디션 성격이면 필요할 수 있어 샘플 확인. |
| `g5_write_new_calendar`, `g5_write_new_calendar02` | 일정/캘린더 성격. 현재 IA에 없으므로 보류. |
| `g5_write_2021_online`, `g5_write_bebe`, `g5_write_mojib`, `g5_write_victory*` | 센터별 특수 게시판. 샘플 확인 후 필요 시 별도 컬렉션. |
| `g5_write_qr_*` | QR/랜딩용 특수 게시판으로 보이며, 공개 사이트 반영 여부 확인 필요. |
| `g5_write_reservation` | 예약/신청 성격. 개인정보 가능성이 있어 보류. |
| `sm_customer` | 고객/상담 성격. 개인정보 가능성이 있어 보류. |
| `g5_member` | 회원 계정. 공개 콘텐츠 이관 범위 밖. |

## 폐기 후보

아래 테이블은 초기 이관에서 버려도 될 가능성이 높다. 단, 삭제 전에는 원본 dump가 보존되어 있어야 한다.

| 테이블군 | 예시 | 이유 |
| --- | --- | --- |
| 방문/통계 | `g5_visit`, `g5_visit_sum`, `g5_popular` | 새 사이트 운영 지표로 승계하지 않는다. |
| 임시/자동저장 | `g5_autosave`, `g5_uniqid` | 마이그레이션 대상 아님. |
| 회원 활동/포인트 | `g5_point`, `g5_login`, `g5_scrap`, `g5_memo` | 공개 콘텐츠와 무관하고 개인정보 가능성 있음. |
| SMS | `sms5_*` | 현재 범위 밖. 대부분 0건. |
| 인증/권한 | `g5_auth`, `g5_group_member`, `g5_cert_history` | 새 Payload 권한 체계로 대체. |
| 설문/FAQ 빈 테이블 | `g5_poll`, `g5_poll_etc`, `g5_faq`, `g5_qa_content` | 0건 또는 현재 범위 밖. |
| 우편번호 | `zipcode` | 20만 건 규모. 외부 표준 데이터라 승계 불필요. |
| 백업 테이블 | `*_bak` | 본 테이블과 비교할 때만 사용. 기본 이관 제외. |
| 빈 게시판 | row 수 0인 `g5_write_*` | 이관 가치 없음. |

삭제 실행 결과:

- 2026-04-21 로컬 MariaDB 복원본에서 위 폐기 후보 조건에 맞는 테이블을 삭제했다.
- 삭제 전 테이블 수: `baewoo` 135, `bnbhighteen` 133, `bnbuniv` 122, `kidscenter` 112
- 1차 삭제 후 테이블 수: `baewoo` 95, `bnbhighteen` 96, `bnbuniv` 82, `kidscenter` 81
- 추가 삭제 후 테이블 수: `baewoo` 83, `bnbhighteen` 84, `bnbuniv` 70, `kidscenter` 71
- 같은 조건으로 재조회했을 때 남은 폐기 후보 테이블은 0개다.
- 원본 dump는 `data/legacy_dumps`에 남아 있으므로 필요하면 `npm run legacy:db:import:reset`으로 재복원한다.

## 통합 실행 메모

- 2026-04-21 `g5_agency`는 `bnb_legacy_work.agencies`로 통합했다.
- 원본 합계는 245건이고, `TRIM(subject)` 기준 고유 항목은 78건이다.
- 통합 결과도 78건이며 중복 `subject`는 0건이다.
- 본문/세부 필드가 완전히 같지는 않으므로 대표 row는 `baewoo`, `kidscenter`, `bnbuniv`, `bnbhighteen` 순서로 선택하고 모든 출처는 `legacy_meta.sources`에 남긴다.
- 재생성 명령은 `npm run legacy:work:agencies`다.

## GUI 확인 순서

GUI로 볼 때는 아래 순서대로 보면 된다.

1. 각 DB의 `g5_teacher`, `g5_teacher2`, `g5_teacher_file`
2. `g5_write_new_notice`, `g5_write_notice`, 하이틴 뉴스 계열
3. `g5_write_new_profile` 계열과 `g5_board_file` 조인 관계
4. `g5_agency`, `g5_write_agency*`
5. `g5_write_new_casting*`
6. `g5_banner*`, `g5_mainimg`, `g5_new_win`
7. 작품/활동 계열: `movie`, `drama`, `appear`, `direct`, `shoot`, `lineup`

## 추천 GUI

Mac에서는 아래 중 하나면 충분하다.

- TablePlus: 가장 빠르고 보기 편하다. 유료지만 이 작업에는 효율이 좋다.
- DBeaver: 무료이고 강력하지만 UI가 무겁다.
- Sequel Ace: MySQL/MariaDB 전용으로 가볍다.

현재 로컬 접속 정보:

```text
Host: 127.0.0.1
Port: 3307
User: root
Password: root
Databases: baewoo, bnbuniv, kidscenter, bnbhighteen
```
